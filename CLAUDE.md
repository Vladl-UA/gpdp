# CLAUDE.md — GPDP/RNA (Claude Code Desktop, локальная сессия)

Этот файл загружается автоматически при старте сессии в этой папке.
Не нужно ничего объяснять заново — но и не нужно пересказывать HANDOFF.md
целиком: он лежит рядом, прочитай его сам первым делом.

## Первое действие

1. Прочитай `HANDOFF.md`, потом `STATE.md`, потом `ARCHITECTURE.md` —
   **с диска**, не с GitHub. В отличие от чат-сессий (Sonnet/Opus/Fable в
   claude.ai), у тебя есть прямой локальный доступ — эта папка и есть
   репозиторий, самая свежая версия по определению, свежее любого `git pull`
   с GitHub. Читать GitHub имеет смысл только чтобы проверить, не разошлась
   ли эта копия с тем, что запушено (см. «Git» ниже).
2. Не пересказывай Владу содержимое файлов — он их не видел, а не забыл.
   Задай один содержательный вопрос по задаче или начни с плана.

## Чем эта сессия отличается от чат-версии (важно для дисциплины)

В чате (claude.ai) синхронизация с GitHub — единственный способ передать
правки: там нет доступа к диску, поэтому весь протокол §8 в HANDOFF.md
построен вокруг `curl raw.githubusercontent.com` и вставного токена для
записи. **Здесь это не нужно.** У тебя уже есть:
- прямой доступ к файлам — правишь на месте, без выгрузки-закачки;
- терминал с `git`, настроенным на аккаунт Влада — коммитишь и пушишь
  обычными командами, без токена;
- локальный PHP и MariaDB — можешь прогнать `php -l file.php` и **прямо
  запустить `php smoke_test.php` в терминале**, не через браузер вообще.

Это отменяет ручной цикл «правлю → выгружаю → прошу Влада залить» —
самую медленную часть предыдущих сессий. Пользуйся этим.

## Дисциплина (не меняется от смены поверхности)

Всё из HANDOFF.md/ARCHITECTURE.md действует одинаково, кто бы ни работал:
- **Архитектура до кода.** Решение, меняющее модель/конвейер/доктрину —
  сначала записать в STATE.md (журнал + пункт «Сейчас»), потом код.
  Не наоборот, даже если код очевиден и быстр.
- **§15 ARCHITECTURE.md** — контрольные вопросы перед архитектурным
  предложением. Не пропускать ради скорости, которую даёт локальный доступ.
- **Naming-Driven Behavior** — имя поля есть адрес; не создавать обходов,
  когда конвейер чего-то не умеет — называть развилку Владу.
- **STATE.md журнал** — append-only, решение фиксируется сразу после
  принятия, не откладывается «допишу потом».
- Разделение админ-файлов (`index.php`/`labels.php`/`configurator.php`) —
  временный фильтр доступа по ролям, не дублирование. Не сливать (журнал
  2026-07-11).

## Git — конкретно

- `main` — единственная боевая ветка, `legacy` — архив старого кода,
  не трогать.
- Коммит-сообщение объясняет **зачем**, не только что изменилось (сверься
  со стилем уже сделанных коммитов — `git log --oneline -10`).
- Перед началом сессии: `git pull`, чтобы не разойтись с тем, что могло
  прилететь из чат-сессии (Sonnet/Opus работали параллельно).
- После правки: коммит + `git push` — не откладывать до конца сессии,
  каждый содержательный шаг отдельным коммитом (тот же принцип, что уже
  соблюдался в GitHub-протоколе чата).

## Локальное окружение

- Сайт: `http://localhost/gpdp/` — можно проверять `curl`'ом из терминала
  или в браузере.
- БД: `gpdp_test`, доступ — см. `config.php` (не печатать пароль в чат/лог).
- Смоук — `php smoke_test.php` из папки проекта; при первом прогоне сам
  создаёт/переполняет изолированную фикстуру `voc_smoke` (журнал 07-12),
  боевых данных не касается.

## Уточнение по словарям, контекстным таблицам и m2m в GPDP/RNA

Важно не сводить автоматически любую структуру вида «A + B + параметр» к generic m2m / junction table.

В GPDP/RNA перед выбором физического хранения сначала нужно определить смысловую роль таблицы в модели:

1.  простой словарь;
2.  зависимая таблица;
3.  таблица состава / рецептуры / контекстного применения;
4.  таблица, приводимая к словарю;
5.  filtered/projected dictionary;
6.  отчётная/представительная выборка;
7.  настоящая m2m-связь;
8.  m2m с payload, если payload действительно принадлежит факту связи.

Ключевая идея:

**Словарь хранит идентичность элемента. Контекстная таблица хранит роль элемента и параметры его применения. Производная выборка из контекстной таблицы может сама использоваться как словарь в другом месте системы.**

Словарь в GPDP/RNA — это не обязательно простая таблица `id → name`. К словарному виду можно привести почти любую таблицу, view, отчёт или выборку, если она может быть представлена как:

```
id
name / label
optional context attributes
```

То есть словарём может быть:

```
voc_material
voc_reagent
voc_buffer
```

но также:

```
reagents_for_material(material_id)
materials_used_in_well(well_id)
materials_available_for_stage(stage_id)
akc_materials_for_horizon(horizon_id)
```

Такие словари могут быть не физическими таблицами, а projected/filtered dictionaries, построенными из контекстных таблиц.

## Пример 1: материал + реагент + процент

Не надо автоматически понимать это как:

```
material ↔ reagent + payload(percent)
```

Более точная смысловая модель:

```
Материал → состав → строка состава
```

Где:

```
reagent = словарный элемент
material = контекст / носитель рецептуры
percent = параметр участия реагента в составе конкретного материала
```

Процент не является свойством реагента глобально.

НТФ сам по себе не «0.5%».  
НТФ становится «0.5%» только в составе конкретного материала / рецептуры / лабораторной записи.

Правильная формула:

```
voc_reagent:
  id
  name

material_composition:
  material_id
  reagent_id
  percent
```

Но `material_composition` здесь не обязательно generic junction table. Семантически это таблица состава / рецептуры / контекстного применения.

Из неё можно получить производный словарь:

```
reagents_for_material(material_id):
  id = reagent_id
  name = reagent_name
  percent = composition_percent
```

И этот projected dictionary можно использовать в другом поле, форме, паспорте горизонта, АКЦ или отчёте.

То есть материал в форме цементирования — это не обязательно прямой `voc_material`. Это может быть projected dictionary из более сложной модели.

## Пример 3: буфер + объём

Буфер тоже не обязательно m2m.

Если в строке цементирования есть:

```
буфер = МБП-С
объём = 3 м³
```

то это можно понимать как контекстное применение словарного элемента:

```
BufferApplication:
  dep_cementing_application
  buffer_id
  volume
```

Или, если буфер всегда один, как поля самой записи цементирования:

```
voc_buffer
data_buffer_volume
```

Если буферов несколько, это повторяемая группа применения буферов:

```
CementingBufferLine:
  dep_cementing_application
  buffer_id
  volume
  weight
```

Это не generic m2m по смыслу. Это строка применения буфера в конкретной операции.

* * *

## Пример 4: реагенты материала как источник словаря

Если есть таблица:

```
material_composition:
  material_id
  reagent_id
  percent
```

то из неё можно получить словарь:

```
reagents_for_material(material_id):
  reagent_id
  reagent_name
  percent
```

Этот словарь можно подставить в другое поле модели.

Например, если в паспорте горизонта или в АКЦ нужно выбрать реагент, связанный с материалом, система не обязана обращаться к полному `voc_reagent`. Она может использовать контекстный словарь:

```
только реагенты, применимые к выбранному материалу
```

И вместе с названием реагента уже доступен процент из состава.

* * *

## Пример 5: АКЦ и материалы

В Листе 4 АКЦ может использовать материалы, уже встречающиеся в цементировании или составе.

Не обязательно строить новую связь:

```
horizon ↔ material
```

Сначала надо проверить, не является ли нужный набор материалов производным словарём:

```
materials_used_in_well(well_id)
materials_used_in_stage(stage_id)
materials_for_horizon_context(horizon_id)
```

Если поле АКЦ должно выбирать материал из уже применённых в скважине, тогда это не новая самостоятельная m2m, а словарь, построенный из контекстной таблицы цементирования.

* * *

## Главное правило классификации

Перед тем как создавать junction table, нужно задать вопросы:

```
1. Это самостоятельная связь двух равноправных сущностей?
2. Или это применение словарного элемента в контексте?
3. Или это строка состава / рецептуры?
4. Или это зависимая таблица от родительской записи?
5. Или это таблица, из которой надо получить словарь?
6. Или это отчётная/представительная проекция?
7. Есть ли у связи собственное поведение, которое требует отдельной физической таблицы?
8. Нужна ли строгая реляционная целостность именно на уровне БД?
9. Нужны ли тяжёлые отчёты/фильтры по этой связи?
```

Generic m2m / junction table выбирается только после этой классификации, а не до неё.

* * *

## Storage plan

Физическое хранение выбирается после смысловой классификации.

Возможные варианты:

```
1. simple voc table
2. text_list field
3. serialized pair list
4. JSON/list field
5. dependent table
6. composition/context table
7. projected dictionary / view
8. true junction table
```

Для простого множественного выбора без payload можно рассматривать `text_list`:

```
,2,3,4,
```

Для повторяемых пар можно рассматривать сериализованный список:

```
10:0.5;25:1.0
```

Для состава, отчётов, фильтров и строгой целостности лучше использовать контекстную/dependent таблицу.

Но даже если физически появляется таблица с двумя id и параметром, её не надо автоматически называть generic m2m. Сначала нужно назвать её смысловую роль:

```
composition table
application table
context table
derived dictionary source
dependent table
junction table
```

## Формула для GPDP/RNA

Не усреднять предметную модель до generic m2m раньше времени.

Сначала определить:

```
что является словарным элементом;
что является контекстом применения;
что является параметром применения;
что может быть приведено к словарю;
что является отчётом/проекцией;
и только потом — как это хранить физически.
```

Для GPDP/RNA это критично, потому что система должна не просто хранить данные, а формировать структуру БД под конкретную модель данных.

Если слишком рано назвать таблицу «связующей», можно потерять её более важную роль: быть таблицей состава, контекстом применения или источником производного словаря для другой части системы.

Короткая версия для вставки в начало задачи:

Правило для m2m/словарей:

В GPDP/RNA не считать автоматически структуру «A + B + параметр» generic m2m. Сначала определить смысл: это связь, состав, контекст применения, зависимая таблица, источник словаря или отчётная проекция?

Словарь — это не только `id → name`. К словарному виду можно привести почти любую таблицу, view или выборку. Контекстная таблица может хранить роль и параметры словарного элемента, а затем сама становиться источником projected/filtered dictionary для другой формы или отчёта.

Пример: `material + reagent + percent` — это прежде всего строка состава материала. Реагент — словарный элемент, процент — параметр его применения в составе конкретного материала. Из таблицы состава можно получить словарь `reagents_for_material(material_id)` и использовать его в других полях модели. Это не обязательно generic m2m.

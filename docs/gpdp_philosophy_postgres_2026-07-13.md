# Размышления о PostgreSQL в контексте архитектуры GPDP

Источник: диалог с Chat, приведён Владом 2026-07-13, второй философский
вопрос дня (первый — GPDP_философия.md, два уровня истины). Оценка и
решение по итогам — в STATE.md, журнал 2026-07-13.

Мы подробно обсудили возможный переход GPDP с MySQL на PostgreSQL.
Просьба оценивать не преимущества PostgreSQL как СУБД (они хорошо
известны), а влияние этих преимуществ на архитектуру GPDP.

## Главный вывод

По состоянию на текущую архитектуру переход на PostgreSQL **не меняет
архитектуру GPDP**. Он может изменить только уровень реализации
некоторых сущностей.

То есть:

```
GPDP
    ↓
Сущность
    ↓
Адаптер хранения
    ↓
MySQL / PostgreSQL
```

При использовании PostgreSQL часть логики отдельных сущностей может
быть перенесена в возможности самой СУБД: DOMAIN, CHECK, пользовательские
типы, RANGE, JSONB, встроенные преобразования, специализированные
индексы.

Но при этом не должны измениться: паспорта сущностей, pipeline,
renderer, модель данных GPDP, контракты сущностей.

Меняется только уровень, на котором выполняется часть работы.

Например, сущность `date` сегодня самостоятельно выполняет валидацию
и преобразование. В PostgreSQL часть этой работы может выполнять сама
база. Однако поведение сущности GPDP не меняется. С точки зрения
архитектуры это лишь оптимизация реализации.

## Важный архитектурный критерий

Предлагается следующий критерий оценки любых возможностей конкретной
СУБД.

**Если использование возможности PostgreSQL изменяет архитектуру GPDP,
значит решение выбрано неправильно.**

Если же возможность PostgreSQL лишь позволяет реализовать существующий
контракт быстрее, компактнее или надежнее, то это допустимая оптимизация.

То есть любые специфические возможности СУБД должны рассматриваться
как свойства адаптера хранения, а не как свойства модели GPDP.

## Разделение преимуществ PostgreSQL

### 1. Архитектурные преимущества

То, что изменяет модель GPDP. На сегодняшний день практически не
найдено возможностей PostgreSQL, которые должны менять архитектуру
ядра.

### 2. Инженерные преимущества

То, что позволяет реализовать уже существующую архитектуру более
эффективно: DOMAIN, RANGE, JSONB, MATERIALIZED VIEW, GiST, GIN,
полнотекстовый поиск, пользовательские индексы, расширяемость
PostgreSQL. Именно эта категория выглядит наиболее интересной.

## GPDP — уровень выше СУБД

В PostgreSQL фундаментальной единицей является **тип данных**. В GPDP
фундаментальной единицей является **сущность**. Однако сущность GPDP
описывает значительно более широкий жизненный цикл: хранение, ввод,
вывод, валидацию, отображение, конфигурирование, связи, поведение в
различных режимах.

Получается, что сущность GPDP является абстракцией более высокого
уровня, чем тип PostgreSQL. Следовательно PostgreSQL не может заменить
сущности GPDP. Он способен лишь предоставить более удобную платформу
для реализации отдельных частей их поведения.

## Возможный архитектурный принцип

> Ядро GPDP не должно знать о существовании конкретной СУБД. Оно
> должно знать только о контракте адаптера хранения.

```
                 GPDP

            Storage Adapter

      MySQL    PostgreSQL    SQLite
```

Любые специфические возможности PostgreSQL используются исключительно
внутри адаптера или реализации отдельных сущностей и не становятся
частью архитектурной модели GPDP.

## Единственная область, заслуживающая отдельного исследования

Практически единственное направление, где PostgreSQL может дать не
только оптимизацию, но и качественное улучшение реализации, — это его
расширяемая система индексов и операторов (GiST, GIN, BRIN, operator
classes и др.). Однако и здесь речь должна идти исключительно о
повышении эффективности реализации существующих сущностей, а не об
изменении архитектуры GPDP.

## Итог

На данный момент складывается впечатление, что архитектура GPDP уже
находится на уровне абстракции выше конкретной СУБД. Поэтому PostgreSQL
следует рассматривать не как замену архитектурных механизмов GPDP,
а как более богатую платформу реализации, отдельные возможности которой
могут использоваться без изменения архитектурных принципов системы.

## Сверка и решение (Sonnet + Влад, 2026-07-13)

Тезис Chat про критерий («фича СУБД не должна менять архитектуру») —
верный, независимое подтверждение: `entities.php` (вся предметная
логика) не касается БД вообще, ни разу — проверено по коду в тот же
день. Схема «Storage Adapter» — НЕ факт, цель: по коду ~150 мест
напрямую зовут `mysqli_*`, без посредника.

Влад заподозрил завышенность списка «инженерных преимуществ» —
справедливо: DOMAIN/RANGE/индексы — действительно одна команда, смена
типа в паспорте. JSONB — локальная доработка паспорта, не архитектура.
MATERIALIZED VIEW — НЕ «одна команда», реальный инженерный вопрос
(когда обновлять) — Chat подал ровнее, чем есть на практике.

**Решение:** делаем простую обёртку вызовов БД (техдолг сам по себе
ценен, не только ради Postgres) — приоритет следующей сессии, ПЕРЕД
остальным. НЕ переходим на Postgres сейчас — до появления настоящего
потребителя (слой отчётов), не «на всякий случай». Точный масштаб —
STATE.md, журнал 2026-07-13.

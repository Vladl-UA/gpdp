## Сверка с кодом и оценка (Sonnet, 2026-07-12)

**Сверено фактом, не на слово:** `bul_`, `sys_`/timestamp-семейство,
файл/изображение — сущностей действительно нет в `entities.php` (полный
список на 12.07: `data`, `voc`, `ltext`, `footnote`, `date`, `year`,
`time`, `int`, `dec`, `calc`). `STRUCTURAL_FIELD_NAMES = ['id', 'rel_main',
'active']` + префикс `dep_` — структурные поля описаны верно.

**Метод — качественный.** Документ читает реальные файлы (ссылки на
строки проверены выборочно, совпадают), честно разделяет «уже работает» /
«не хватает», не предлагает ни одного обходного пути в ядре под
конкретное приложение. Финальный критерий («если для взаимного like
понадобится писать `dating_find_matches()` в коде — проверка провалена»)
— переформулировка нашего же принципа от противного.

**Заявленная «главная находка» — не то, чем является на самом деле.**
Документ подаёт «сеть равноправных объектов, не дерево» как открытие.
Но большинство «сетевых» случаев (`profile_like`, участники `match`) он
же сам разворачивает в обычные `dep_`-таблицы — это не новый класс.
Настоящая новая мысль — глубже, в §6.1: полю нужен явный адрес источника,
не выводимый из имени (`voc_birth_city`/`voc_current_city` → оба `city`).
Это НЕЗАВИСИМО совпало с примером Влада в этом же разговоре (два цвета
из одного словаря → идея `link_`, журнал 07-12) — конвергенция двух
независимых разборов на одной находке, сильный сигнал, что она настоящая.

**§6.2 (универсальная проекция/отчёты) переоценена по месту в очереди.**
Формально нужна, но по объёму — вторая платформа внутри платформы, не
«между делом». Совпадает с уже зарезервированной пустой полкой «Отчёты»
в конфигураторе (`render_table_directory`, 07-11/12) — место есть,
наполнение не сейчас.

Выжимка принятых пунктов и их приоритет — в STATE.md, раздел «Позже»,
запись «Стресс-тест: сайт знакомств».

## Полный текст документа Chat

Ниже — проект модели по состоянию ветки `main` репозитория GPDP на 12
июля 2026 года. Сайт знакомств рассматривается как проверочная
конфигурация платформы, а не как отдельное приложение, для которого
допустимо дописывать предметный код. Это соответствует исходной цели
GPDP: модель хранится в БД, имя поля выбирает процедурную сущность, а
приложение возникает посредством конфигурации.

## 1. Исходные ограничения

Из рассмотрения исключены: регистрация, пароли, сессии и восстановление
доступа; разграничение прав; защита персональных данных; антиспам,
модерация и административные расследования; платёжная инфраструктура.

Предполагается, что прикладная модель уже получает идентификатор
текущего профиля из будущего доверенного пользовательского контура.

Проверяется именно функциональная способность GPDP описать: анкету;
фотографии и многозначные характеристики; предпочтения поиска; выдачу
подходящих анкет; лайки, пропуски, избранное и блокировки; взаимную
симпатию; переписку; историю взаимодействий.

Текущее ядро уже поддерживает обычные поля, словари, длинный текст,
даты, время, целые и дробные числа. Оно также умеет строить дерево
владения через `dep_`, карточки объектов, формы и стандартный CRUD.

## 2. Основная модель данных

### 2.1. Словари

Большая часть классификационных характеристик должна оставаться
обычными словарями:

| Таблица | Назначение |
|---|---|
| `voc_gender` | Пол или способ самоопределения |
| `voc_relationship_goal` | Дружба, отношения, брак и т. п. |
| `voc_interest` | Интересы |
| `voc_language` | Языки |
| `voc_language_level` | Уровень владения языком |
| `voc_city` | Населённые пункты |
| `voc_education` | Образование |
| `voc_smoking` | Отношение к курению |
| `voc_alcohol` | Отношение к алкоголю |
| `voc_children_status` | Наличие детей и отношение к детям |
| `voc_profile_status` | Черновик, опубликован, скрыт |
| `voc_match_status` | Активно, завершено |

Это простые склады идентичностей. Сведения вроде «насколько важен
интерес» или «каким языком и на каком уровне владеет человек» принадлежат
уже не словарю, а контекстной строке использования. Словарь даёт
элемент, контекстная таблица — его роль и параметры применения.

### 2.2. Профиль

```
profile
├── id
├── data_name
├── date_birth
├── voc_gender
├── voc_city
├── voc_relationship_goal
├── int_height_cm
├── voc_education
├── voc_smoking
├── voc_alcohol
├── voc_children_status
├── ltext_about
└── voc_profile_status
```

`profile` — корневая предметная таблица. `data_name` — отображаемое имя,
одновременно позволяет таблице выполнять словарный контракт адресного
словаря: `voc_profile` в других таблицах сможет разрешаться в записи
профилей. Возраст — не отдельное поле, вычисляется из `date_birth` на
дату запроса.

Почти вся анкета собирается существующими сущностями (`data_`, `date_`,
`voc_`, `int_`, `ltext_`) — базовую анкету GPDP уже способен создать
через конфигуратор без изменения ядра.

### 2.3. Фотографии

```
profile_photo
├── id
├── dep_profile
├── file_image
├── data_caption
├── int_sort_order
└── bul_primary
```

Зависимая таблица профиля. Сейчас нереализуема полноценно: сущности
файла/изображения нет, булевой сущности нет, нет проверки «только одна
главная фотография», generic view не выводит галерею как галерею.

### 2.4. Интересы

```
profile_interest
├── id
├── dep_profile
├── voc_interest
└── int_importance
```

Не generic m2m — контекстная таблица участия интереса в анкете.
Аналогично языки:

```
profile_language
├── id
├── dep_profile
├── voc_language
└── voc_language_level
```

Уже укладывается в существующий механизм `dep_profile + voc_*`.

### 2.5. Предпочтения поиска

```
profile_preference
├── id
├── dep_profile
├── int_age_from
├── int_age_to
├── int_height_from
├── int_height_to
├── dec_distance_km
├── voc_relationship_goal
└── bul_only_with_photo
```

Многозначные предпочтения — отдельными таблицами:

```
profile_preferred_gender
├── id
├── dep_profile
└── voc_gender

profile_preferred_interest
├── dep_profile
└── voc_interest
```

Хранение и применение предпочтений — разные задачи; GPDP уже может
сохранить строки, но не умеет превратить их в параметризованную выборку.

## 3. Взаимодействия между профилями

### 3.1. Направленные действия

Для первой версии — не одна абстрактная таблица, у действий разная
семантика и правила жизненного цикла:

```
profile_like
├── id
├── dep_profile   ← кто поставил лайк
├── voc_profile   ← кому поставлен лайк
└── sys_created_at
```

По той же схеме: `profile_pass`, `profile_favorite`, `profile_block`,
`profile_view`.

Многие внешне «многие-ко-многим» отношения по-прежнему выражаются
контекстными таблицами — generic `ent_mlt` для них не обязателен.

**Ограничение нынешней системы:** карточка профиля покажет исходящие
действия как детей по `dep_profile`. Полученные лайки — обратная выборка
(`profile_like.voc_profile = текущий профиль`), а граф отношений
компилируется только по `dep_`; обычные словарные ссылки обратного графа
не создают. Структурным префиксом сейчас является только `dep_`, `rel_main`
выделен отдельным именем. Следовательно: «кого я лайкнул» — видно через
дерево; «кто лайкнул меня», «кто посмотрел», «кто добавил в избранное» —
generic CRUD сейчас не показывает. Сайт знакомств сразу требует
универсальных обратных выборок, не только дерева владения.

### 3.2. Взаимная симпатия

Match логически определяется так: A поставил like B, и B поставил like
A, и между ними нет block — первично match является производным
фактом, не ручной таблицей. После возникновения полезно материализовать
объект:

```
match
├── id
├── voc_match_status
└── sys_created_at

match_member
├── id
├── dep_match
└── voc_profile
```

У активного match — ровно две строки `match_member`. Лучше таблицы с
`voc_profile_1`/`voc_profile_2`: участники получают собственную
контекстную таблицу, роли симметричны, не нужно угадывать «первого» и
«второго», модель пригодна для расширения.

Текущая платформа не может автоматически: обнаружить взаимный like,
проверить отсутствие block, создать match, добавить двух участников,
сделать всё одной транзакцией, не создать второй такой же match. Это
уже не CRUD поля, а правило процесса.

## 4. Переписка

Отдельный `conversation` не обязателен — `match` может быть контейнером
диалога:

```
message
├── id
├── dep_match
├── voc_profile   ← автор сообщения
├── ltext_body
└── sys_created_at

message_read
├── id
├── dep_message
├── voc_profile
└── sys_read_at
```

Хранение текста сущностью `ltext_` уже возможно. Отсутствуют: timestamp;
сортировка по времени; постраничная загрузка; выборка сообщений
участника; проверка, что автор входит в `match_member`; счётчик
непрочитанных; автосоздание `message_read`; контекстная форма «отправить
в этот match».

## 5. Выдача анкет

Главный экран — не список таблицы `profile`, а параметризованная
проекция `candidate_feed(current_profile)`: исключить себя; исключить
скрытые; исключить заблокированных в обе стороны; исключить просмотренные
недавно; применить возраст, пол, цель, расстояние; учесть многозначные
интересы; отсортировать; вернуть порциями. Это наиболее существенная
недостающая часть. Параметризованные словари, отчёты, выборки,
batch-режимы и правила валидации уровня записи уже отложены в дорожной
карте — для сайта знакомств они становятся центральным слоем, не поздним
улучшением.

## 6. Что сайт знакомств обнаруживает в GPDP

### 6.1. Контекстная ссылка на произвольный источник

`voc_x` жёстко адресует таблицу `voc_x` либо `x` — работает для
`voc_profile → profile`. Но в одной таблице нельзя иметь две ссылки на
один источник под разными ролями: «город рождения» и «нынешний город»
оба должны вести к `city`, «инициатор» и «получатель» — оба к `profile`.
Два поля `voc_city` физически невозможны. Нужен модельный параметр
конкретного поля:

```
поле: voc_birth_city
source: city
role: birth_city
```

Не должно записываться в имя и не должно задаваться SQL-текстом.
Естественное место — будущий `field_config` или паспорт адресной
проекции. **Первый критический недостающий элемент** — не новая сущность
предметной области, а универсальная способность одному полю иметь
собственный скомпилированный адрес источника.

### 6.2. Параметризованные выборки и проекции

Нужен общий механизм: источник; JOIN-пути; фильтры; параметры из
доверенного контекста; сортировка; лимит и пагинация; возвращаемые поля;
шаблон карточки; агрегаты. Потребуется для: списка кандидатов, входящих
лайков, взаимных лайков, списка matches, непрочитанных, общих интересов,
расстояния, истории просмотров. Существующий словарный компилятор
показывает правильное направление (проверка ссылок при сборке snapshot,
исполнение плана в runtime), но собирает в основном `id + label`; сайту
знакомств нужна полноценная выборка записей. **Второй критический
недостающий элемент** — фактически следующий необходимый крупный слой.

### 6.3. Правила процесса

Универсальные реакции на событие: после создания like — найти обратный,
проверить отсутствие block, если match отсутствует — создать match и
двух участников. После block — исключить из выдачи, запретить сообщения,
возможно закрыть match. После удаления главной фотографии — назначить
следующую. Не handler отдельного поля — правило охватывает несколько
строк и таблиц. Нужны: событие, условие, последовательность операций,
транзакция, идемпотентность, структурированный результат. **GPDP
требуется слой модельных операций/workflow**, отдельный от сущностей
полей — не должен превращаться в предметный PHP-код вроде
`create_dating_match()`, правило задаётся конфигурацией.

### 6.4. Валидация уровня записи и связи

`age_from <= age_to`; `height_from <= height_to`; `like.actor != like.target`;
match имеет ровно двух участников, участники различны; `message.author`
принадлежит match; у профиля одна primary photo; нельзя одновременно
активные like и block к тому же профилю. Подтверждает уже обозначенную
в STATE.md потребность в правилах валидации уровня записи и связи. Нужен
отдельный слой: field validation, row validation, relation validation,
operation validation — нельзя смешивать в handler одного случайно
выбранного поля.

### 6.5. Ограничения и индексы

Конфигуратор создаёт поля nullable; FK и обязательность для `dep_`
сознательно отложены. Понадобятся: `NOT NULL`, внешний ключ, `UNIQUE`,
составной `UNIQUE`, индексы, возможно `ON DELETE CASCADE`. Примеры:
`UNIQUE(profile_like.dep_profile, profile_like.voc_profile)`,
`UNIQUE(match_member.dep_match, match_member.voc_profile)`,
`INDEX(profile_like.voc_profile)`,
`INDEX(message.dep_match, message.sys_created_at)`. Без них лайки могут
дублироваться, участник — дважды входить в match, обратные выборки
станут дорогими. Паспорту поля недостаточно описывать только DB type —
конфигуратору нужен декларативный механизм ограничений таблицы и группы
полей.

### 6.6. Системное время

Нужно почти повсюду: создание анкеты, изменение, последний визит, like,
просмотр, match, сообщение, прочтение, блокировка. Обычные `date_`/`time_`
не заменяют timestamp — это пользовательские календарные значения, не
автоматически проставляемые события. Нужна структурная сущность/семейство
`sys_created_at`/`sys_updated_at`/`sys_last_seen_at` с автоматическими
`DEFAULT CURRENT_TIMESTAMP`/`ON UPDATE`. Уже отмечено в репозитории как
открытый вопрос. **Timestamp — обязательный элемент ближайшего этапа, не
далёкий аудит.**

### 6.7. Файл и изображение

Нужна сущность, возвращающая storage plan, не пытающаяся записать файл
как значение обычной колонки: validate mime/size → storage operation →
сохранить идентификатор/путь → структурированный результат.
Дополнительно: несколько размеров, поворот, порядок, главная фотография,
удаление физического файла после подтверждённого удаления записи.
Контракт storage plan уже предусмотрен архитектурно, но не реализован.

### 6.8. Булево поле

Постоянно требуется: главная фотография, показывать возраст, только с
фото, уведомлять о сообщениях, скрыть профиль. Отдельный словарь «да/нет»
для каждого случая — искусственно. Нужна обычная сущность `bul_x` с
checkbox в формах и `TINYINT(1)` в БД.

### 6.9. Географическое расстояние

`voc_city` достаточно для фильтра по городу, не для расстояния.
Минимальная модель — `dec_latitude`/`dec_longitude` на `profile`. Для
выдачи понадобится вычисление расстояния и сортировка по нему: формула,
функция проекции, фильтр по вычисленному значению, возможно специальный
географический тип позже. На первом этапе отдельная geo-сущность не
обязательна — достаточно допустить безопасную типовую функцию в плане
выборки.

### 6.10. Разные представления одного объекта

Один `profile` отображается по-разному: полная анкета, редактирование,
публичная карточка, карточка в выдаче, краткая строка в likes, участник
match, автор сообщения. Один generic `record_table_view` этого не
выразит. Нужен form/view contract: `profile.edit`, `profile.public`,
`profile.discovery_card`, `profile.compact` — соответствует отложенной
идее `_form_id`, сайт знакомств делает её реальным потребителем, не
теоретическим расширением.

## 7. Что можно собрать уже сейчас

Без изменения ядра: `profile` с `profile_interest`, `profile_language`,
`profile_preference`, `profile_preferred_gender`, `profile_like`,
`profile_pass`, `profile_favorite`, `profile_block`; `match` с
`match_member` и `message`. Можно проверить: создание анкет, словарные
поля, многозначные характеристики через дочерние таблицы, исходящие
лайки, ручное создание match, ручное добавление участников, хранение
сообщений. **Структурный скелет предметной области GPDP уже выдерживает.**

Не заработают как законченный сайт: загрузка фотографий, автоматическая
выдача, входящие реакции, определение взаимного like, автосоздание
match, корректная переписка с непрочитанными, ограничения уникальности,
сортировка и пагинация, полноценные карточки разных видов.

## 8. Главный архитектурный результат

Сайт знакомств не доказывает срочную необходимость generic m2m —
интересы, языки, реакции и участники match естественно описываются
контекстными зависимыми таблицами; доктрина подтверждается. Но модель
выявляет класс, который тампонажная задача почти не затрагивала: сеть
равноправных объектов и событий между ними. Там, где тампонаж строится
как дерево (скважина → ступень → интервал → состав), сайт знакомств
строится как граф (профиль → действие → другой профиль; профиль ↔ match
↔ профиль; match → сообщение → автор-профиль). Существующего дерева
`dep_` для этого недостаточно. Нужны не новые предметные костыли, а
универсальные механизмы: ссылка с ролью на произвольный источник;
обратные и параметризованные выборки; правила процесса; валидация записи
и связи; составные ограничения и индексы; timestamp; file/image; bool;
контекстные представления.

**Приоритет реализации** (предложение Chat, после завершения нынешнего
базового этапа): 1) `sys_timestamp`; 2) `bul`; 3) параметры поля и
ограничения (nullable, unique, index, FK); 4) адрес источника конкретного
`voc_`-поля; 5) универсальная проекция с фильтрами/сортировкой/параметрами
контекста; 6) row/relation validation; 7) модельные операции и
транзакционные workflow; 8) file/image; 9) form/view contracts;
10) географические и расчётные функции.

**Итоговый критерий:** если после реализации перечисленного сайт
знакомств можно получить созданием таблиц, словарей, проекций, правил и
представлений через конфигуратор — GPDP пройдёт проверку. Если для
взаимного like или списка входящих симпатий придётся написать в ядре
функции вроде `dating_find_matches()`, `dating_received_likes()`,
`dating_candidate_feed()` — предметная модель просочилась в код и
проверка провалена.

   CREATE TABLE model_registry (
     id           INT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY,
     data_kind    VARCHAR(16)  NOT NULL,   -- 'table' | 'field'
     data_owner   VARCHAR(64)  NULL,       -- NULL для kind='table'
     data_element VARCHAR(64)  NOT NULL,
     active       TINYINT(1)   NOT NULL DEFAULT 1,
     UNIQUE KEY uq_address (data_kind, data_owner, data_element)
   ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_general_ci;

     dep_model_registry INT UNSIGNED NOT NULL PRIMARY KEY,
     data_short         VARCHAR(64)  NULL,

2. **Семейство времени — ВЫПОЛНЕНО 2026-07-05**: `ent_date`, `ent_year`,
   `ent_time` живы на целевом localhost, ядро (core.php) не тронуто ни
   строчкой. Честная поправка к исходному плану: composite-input
   (day+month+year отдельными ключами POST) НЕ понадобился — каждая из
   трёх сущностей нашла ровно одно нативное значение и один нативный
   HTML-виджет (`<input type="date/number/time">`), сборки из частей
   не потребовалось вообще. Composite-механика (конвенция `field__part`,
   разбор в record_save) откладывается до первой сущности, у которой
   действительно нет родного виджета — кандидат `ent_dur` (сроки
   схватывания «6-40», MySQL TIME физически годится, HTML type="time"
   — нет, это «время суток», не «сколько прошло»). `ent_year`/`ent_time`
   построены без текущего предметного поля, по прямой директиве
   «типы под всё, что поддерживает БД» — не спекуляция, осознанная
   политика полноты семейства (тот же приём, что с `footnote`/`ltext`).
   `ent_year` хранится как `smallint`, не MySQL `YEAR` — последний узок
   (1901–2155) без реальной причины его придерживаться; validate
   проверяет границы ВЫБРАННОГО типа, не выдуманную календарную
   семантику. Найден и исправлен реальный класс бага: `nullable=true`
   в паспорте не означает, что validate обязан принимать пустую строку
   как валидное значение для DATE/YEAR/TIME (в отличие от varchar,
   где '' законно) — пустая строка конвертируется в NULL по ЖИВОЙ
   схеме колонки (`field['schema']['nullable']`), не по дефолту паспорта.
   Первый предметный кейс: дата в ТЗ-2 лежит Excel-сериалом (39604) —
   конвертация при построении тестовых данных, ещё не выполнена.
3. **Зависимые таблицы (`rel_`/`dep_`) — ВЫПОЛНЕНО 2026-07-08.** Несущая
   конструкция референсной задачи. Семантика прояснена по легаси-дампу
   `tt_utf8_fixed.sql` (археология смысла, не образец кода):
   - **`dep_<table>`** = «dependent on» — FK на строке-потомке →
     непосредственный родитель, ставит ссылающуюся таблицу в подчинение
     (`bufers.dep_material → material`, `material.dep_steps → steps`,
     `steps.dep_main → main`). Это и есть «ссылка на произвольного
     непосредственного родителя» — изобретать не нужно, уже существует.
   - **`rel_main`** = «relates to main» — плоская ссылка с любой глубины
     сразу на корень дерева, параллельно `dep_`. Один из вариантов m2m:
     короткий путь к корню, минуя промежуточные уровни. Каждый потомок
     несёт И свой `dep_` (родитель), И `rel_main` (корень) одновременно.
   - Легаси `find_dep`/`related_records` делали это рантайм-перебором
     всех таблиц (`mysql_list_tables` на каждый чих) — «с кучей запросов,
     зато да». У нас граф компилируется в снапшот ОДИН раз, рантайм не
     ищет (§8).
   - Две корневые (main и well) — артефакт слетевшего тест-режима, НЕ
     норма: корень один, `main` (в дампе все `rel_main`, `steps.dep_main`).

   **Шаг 1 — композитная подпись объекта.** `record_label($db, $snapshot,
   $table, $id)` — подпись САМОЙ записи: три ветки — проекция через
   `lookup_labels` (переиспользован исполнитель а2) → `data_name` → `#id`.
   Попутно закрыт дефект резолвера: таблица с шаблоном, на которую НИКТО
   не ссылается voc-полем, раньше не компилировалась вовсе (подпись
   объекта = `#id`). `snapshot_build_dictionaries` дособирает планы для
   любой таблицы с маяком под ключом `@table` (`@` невозможен в имени
   поля — коллизий нет).

   **Шаг 2 — компиляция графа.** `snapshot_build_relations(structure)` —
   чистая функция над структурой, ноль SHOW. Обратный индекс
   `model.relations[parent] = [{child, fk}]` из `dep_`-полей. `rel_main`
   компилируется в отдельный разрез (`model.relations_root`), НЕ
   используется в рендере («нисходящих достаточно»). Врезано в
   `snapshot_build` + `snapshot_validate`.

   **Шаг 3 — карточка тянет детей.** `record_children($db, $snapshot,
   $table, $id)` — по `model.relations[$table]` один запрос на дочернюю
   таблицу (`WHERE dep_<table> = id`), подпись строк через `record_label`.
   index.php: блоки детей под формой edit + «добавить ребёнка» на `new`
   дочерней с предзаполненным родителем. Единственная точка касания
   записи: предзаполнение `dep_` — через доверенный параметр задания
   (`_parent_table`/`_parent_id`), сверяемый с `model.relations` И
   существованием родительской записи, НЕ сырой `dep_`-инъекцией из
   POST (§9). `record_save` получил структурный канал: **insert-only**
   (на UPDATE игнорируется — reparent остаётся отдельным защищённым
   действием, п.5), с проверкой `kind==='structural'` на каждое поле
   канала — не-структурное поле в канале отклоняется.

   Смоук: секция 1г, 7 проверок (граф на синтетике по топологии
   легаси-дампа, fail-fast на `dep_` в никуда, живые `record_children`/
   `record_label`). Полный HTTP-цикл подтверждён через `php -S`: форма

   redirect в карточку родителя → ребёнок виден в блоке. Все 10 секций
   смоука + 1а-1г зелёные на целевом localhost, 2026-07-08.

   **Два бага найдены и закрыты в процессе (журнал 07-08), оба —
   собственные регрессии дособорки `@table`:** (1) `compile()` в

   на уже проваленном источнике (из дособорки `@table` после провала

   различать «ещё в работе» (цикл) и «завершилось неудачей» (не

6. **Снапшот: тумблер режима + автоперестройка конфигуратором — ВЫПОЛНЕНО 2026-07-08.**
   Автоперестройка конфигуратором оказалась УЖЕ реализованной (часть
   «Конфигуратора v0», 07-05) — просто не была увязана с формулировкой
   п.6: `configurator_create_table`/`delete_table` уже гоняют протокол
   §8 целиком, ручной `rm snapshot.php` после структурных операций не

   `live` в `snapshot_init` — прямой `return snapshot_build(...)` без
   файлового пути, лок проверяется одинаково первым шагом в обеих

   `PHP_BINARY` оказался пуст → `Permission denied`. Правильно — в одном
   процессе, вызовом `snapshot_build()` напрямую (это и есть live-путь
   буквально). Принцип: **проверки смоука живут в одном процессе, без
   subprocess/PHP_BINARY.**

   Исходный контракт (для истории):
   Снимает недельную боль ручного `rm state/snapshot.php`. Два связанных
   шага, оба — тонкий слой поверх уже написанного, не новый механизм:
   - **Тумблер режима в config** — одно значение `snapshot.mode`:
     `cached` (по умолчанию, боевой путь — `include` файла через
     `snapshot_init`, как сейчас) либо `live` (dev-путь — та же
     `snapshot_build()` БЕЗ шага записи; результат живёт один рендер
     в памяти). Функция сборки одна, второго пути не появляется —
     «тонкая труба» цела. Флаг — параметр окружения в системном конфиге,
     **не** request (GET/POST): иначе обычная форма получает рычаг к
     schema-поведению (нарушение §15.4). В `live` staleness исчезает по
     построению — файла нет, нечему протухать/сиротеть. Прод `live`
     не ставит никогда — живая интроспекция как штатный режим прода
     остаётся табу §12.
   - **Конфигуратор сам перестраивает снапшот** в конце структурной
     операции — оформление протокола §8, который всю неделю проходили
     руками: `lock → DDL → запись в model_registry (+labels) →
     snapshot_rebuild_structure (собирает и атомарно пишет temp+rename) →
     unlock`. В `cached` цикл сохраняет файл; в `live` шаг записи пропущен.
     Ручное `rm` уходит из нормального workflow — остаётся только аварийной
     dev-операцией.

   Названная цена: живя в `live`, перестаёшь прогонять сам файловый путь
   (`include`, atomic write, холодный старт), а в проде работает именно он.
   Гарант остаётся один — смоук, всегда строящий + пишущий + читающий
   настоящий файл (секция 1). Пока смоук трогает файл — dev-режим выключает
   неудобство, не выключая покрытие продового пути.
   Остаточный класс (НЕ блокер сегодня): падение конфигуратора между `lock`
   и `unlock` оставляет осиротевший `schema.lock` → 503 (PHP `finally` не
   отрабатывает при fatal/exit, журнал 2026-07-03). Лечение пока ручное
   `rm state/schema.lock`; автоснятие сироты — открытый вопрос, повышенный
   до обязательного перед приходом нескольких писателей (см. открытые).

configurator.php вызывает core.php напрямую (те же snapshot_build_*/

не вложенный контур.

(auth: зарезервировано) → разбор request → prepared task → конвейер →
render | PRG-redirect.

## Требования к развёртыванию (выстрадано 2026-07-04)

- `state/` должен быть доступен на запись ОБОИМ контурам: консоли и
  веб-службе. При сайте в /home — обязателен drop-in службы
  (`ReadWritePaths=<путь>/state`), иначе `ProtectHome=read-only` даёт
  EROFS только вебу («мигающая» ФС — две точки зрения на каталог).
  Чище — размещение вне /home.
- `state/`: владелец-группа единые, setgid на каталоге (`chmod 2775`),
  веб-пользователь в группе. Ручные chmod-«подтяжки» запрещены.
- `pcre.jit=0` в ini обоих SAPI (JIT упирается в MemoryDenyWriteExecute).
- Ручные прогоны — всегда от одного пользователя; root — только для
  системных действий.
- Новый localhost = залить эталонный дамп + прогнать смоук.

**Два уровня работы с БД — разведены явно, не смешивать (журнал
2026-07-14, диалог с Chat + сверка).** Раньше в этом списке были две
голые строки («отчёты и выборки», «единая труба исполнения
`sql_execute`») без разграничения — рисковали слиться в одну задачу
при чтении. Это два разных по масштабу уровня:

- **Нижний уровень (уже спланирован, приоритет — см. п.0 «Сейчас»):**
  `db_select`/`db_execute` — механическая обёртка вызова
  (`prepare→bind→execute→fetch` в одну функцию вместо ~150 копий).
  НЕ конструктор SQL, НЕ план выборки — просто убирает дублирование
  вызова. Сущность в это не вовлечена вообще.

- **Верхний уровень (спроектирован по принципу, код НЕ написан) —
  `record_select($plan)`, структурированный план выборки, симметричный
  тому, что `record_save()` уже делает для записи.** Для записи:
  модель таблицы + массив значений = INSERT/UPDATE (уже так, проверено
  кодом — `record_save()` берёт поля из снапшота, значения из входного
  массива по совпадающему имени, один универсальный запрос). Для
  чтения должен появиться симметричный механизм: модель выборки +
  контекст + параметры = SELECT. НЕ готовый SQL, НЕ отдельная
  PHP-функция на каждый отчёт, НЕ универсальная строка SQL в данных —
  план, описывающий СМЫСЛ операции («операция — дети, связь — такая-то,
  id родителя — такой-то»), не текст запроса. Граница ровно та, что
  уже действует в §16.3 («текст SQL в данных не хранится и не
  исполняется») и §15 («не плодит ли абстракцию там, где хватает
  passport handler call»): план с ключом `'sql'`/`'join'`/`'expression'`
  внутри — запрещённый конструктор запросов (второй SQL внутри
  массива); план с ключами `'operation'`/`'relation'`/`'parent_id'`
  (или `'source'`/`'projection'`/`'criteria'`) — компилятор ОГРАНИЧЕННОГО,
  заранее известного набора типовых операций, разрешено.

  Что уже есть (проверено кодом, `record_children()`): типовая форма
  для `dep_` — один шаблон (`SELECT * FROM {child} WHERE {fk} = ?
  ORDER BY id DESC`), имена берутся из скомпилированного графа связей,
  ни одного запроса под конкретную пару таблиц не пишется вручную.
  `record_fetch`/`record_list`/`record_label` — та же дисциплина для
  своих канонических форм. Чего нет: типовой формы для `link_`
  (частично переиспользует словарный механизм, отдельного пути нет —
  это скорее хорошо), `rel_main` (граф компилируется в
  `model.relations_root`, но исполнитель среза не реализован), `m2m`
  (`owner → junction → target`, единой схемы нет), произвольных
  фильтров/диапазонов/сортировки/пагинации/JOIN/агрегатов/группировок.

  **Правило (сформулировано Chat, принято как есть):** ни одна новая
  предметная выборка не получает собственный SQL. Сначала определяется
  её типовая форма (список, связь, проекция, фильтр, агрегация, дерево,
  m2m, отчётный план). Форма реализуется один раз для всей системы.
  Опасность не в текущих `record_list`/`record_fetch`/`record_children`
  (канонические формы доступа, не предметные случаи) — опасность в
  первой функции вида `record_well_cementing_report()`, названной по
  смыслу конкретного экрана, не по типу операции. Такой функции пока
  нет ни одной — граница не пересечена.

  НЕ проектировать до реального повода (тот же принцип, что отложил
  таблицу весов и Postgres): типовые формы по `rel_main`/`m2m`/фильтрам
  добавлять по мере появления настоящего потребителя, не заранее.

**Контекст, роли, фильтр отображения, слияние админ-входов (замысел
2026-07-11).** Контекст = объект + действие + роль (список открыт,
подъедет ещё). На контексте строится фильтр отображения. Роли по
нарастанию прав: простой пользователь (только отчёты — представления
данных; базовые таблицы напрямую не видит) → администратор данных
(управляет данными, видит базовые таблицы) → системный
администратор/конфигуратор (заливка БД, правка модели, почти всё).
Сегодня фильтр эмулируется разделением на файлы (index/labels/
configurator — каждый сам себе фильтр доступа); это временная несущая
подпорка, НЕ дублирование (см. журнал 07-11 — не сливать входы). Когда
дойдём: настоящий фильтр по контексту + слияние входов в один + сами
отчёты (типы таблиц-представлений). Требует пользователей, прав,
безопасности — не проектировать до работающей базы.

**Форма продукта «код-раз, моделей-много»** (осознана 2026-07-07,
проектирование отложено). Один код обслуживает множество БД-моделей,
каждая БД = отдельное приложение; пользователь владеет одной/несколькими
моделями, разные пользователи — разными наборами; версия модели задаётся
на пользователя. Это SaaS-мультитенантность с изоляцией по БД — форма
известная, новизна лишь в наполнении (содержимое приложения в модели
данных, не в темах/плагинах). Три следствия для будущего (не сегодня):
снапшот принадлежит модели, а не инсталляции (идентичность модели
становится **адресом** файла, не полем-маячком); коннект = выбор активной
модели на запрос, а не глобальная константа (одна активная модель за раз
сохраняется — пул/мультиконнект НЕ проектируется); config раздваивается
на системный (где код, кого пускать, как связать пользователя с моделями)
и тенантский (какие модели, какая активна). Красная линия: назначение
модели — доверенный слой окружения, не параметр request (§15.4). Партицио-
нирование «инженер на свой участок БД» — та же группа отложенного. Не
проектировать до окончания базовой фазы (сущности, словари, зависимые
таблицы Листа 2). Точка возврата названа явно — журнал 2026-07-07 —
чтобы будущая сессия не изобретала форму заново.

**`rel_`/`dep_` — уточнённое различение (журнал 2026-07-08, по разбору
легаси param.php/button_new/db_tree, не только дампа).** `dep_<parent>` —
связь ВКЛЮЧЕНИЯ в иерархию (дерево владения): запись — ребёнок parent,
показывается в его карточке, `record_children` читает по этому графу.
`rel_<root>` — связь ПРИНАДЛЕЖНОСТИ корневому досье, ШИРЕ дерева: в
легаси покрывала не только сквозной доступ к корню для записей на
`dep_`-линии, но и **боковые таблицы вне линии** (пласты, замеры,
примечания) — те, что относятся к центральной записи, но не являются
ничьим прямым ребёнком. `record_children` их не видит и не должен;
будущая `record_root_related($db, $snapshot, root_table, root_id)` по
`model.relations_root` — читает и `dep_`-детей, и боковые `rel_`-записи
разом, как «досье» — в «Позже», не сегодня.

**`rel_main` — частный случай `rel_<root>`, не особое имя (уточнение
07-08).** `main` в легаси не был предметным понятием модели — он был
единственным системно-обязательным корнем, поэтому имя корня и имя
механизма совпали, что и создавало иллюзию, будто `rel_main` — кусок
модели данных, зашитый в системное имя. На деле формула та же, что у
`dep_<parent>`: `rel_<root>`, где `root` — имя корневой таблицы, сегодня
всегда `main`, потому что корень один. **Мультикорневость** (несколько
`rel_<root>` в одной модели, например `rel_скважины` наравне с будущим
вторым корнем) — не то же самое, что мультитенантность (много моделей)
из блока выше: это одна модель с несколькими деревьями. Логически та же
группа отложенного, не проектируется до реальной нужды — сегодня Лист2
даёт ровно одно дерево с одним корнем `main`, `rel_main` работает как
есть, код не трогается. Комментарий к `model.relations_root` в
`snapshot_build_relations` — уточнить при следующем касании `core.php`
(не сейчас, отдельным мелким заходом): сегодня написано «плоская ссылка
на корень, минуя уровни» — узко, не отражает боковые таблицы вне
`dep_`-линии.

**Группы логически связанных полей (`от/до`, `план/факт`, `объём/состав/вес`)
— отложено до первого реального потребителя, НЕ отклонено** (обсуждение
2026-07-08, Влад + Чат). Пары/тройки полей — не случайное соседство, а
логические связки: диапазон, план-факт, набор-измерение. Система подписей
(`model_labels`) их НЕ трогает — группировка через подписи запутала бы
презентацию (Чат прав); renderer тоже НЕ угадывает группу по суффиксу
имени — это была бы «магия по имени поля» (§1). Три пути кодирования
рассмотрены:
- **через тип поля** (`startdec`/`stopdec` — Влад): НЕ смешение кода
  и данных, как я сначала ошибочно решил — `start`/`stop` кодирует
  УНИВЕРСАЛЬНУЮ структурную роль «граница диапазона» (глубина, даты,
  цена, температура — любая пара), а не предметное «цементирование».
  Взрыв типов возникает лишь при `startint/startdate/plandec/...` —
  чистая пара `start/stop` как ОДИН паттерн, наоборот, экономит. Идея
  сильная, НЕ отклонена;
- **через модельный слой** (`model.groups`/`composite_field` — Чат):
  snapshot знает `{group: depth_range, type: range, members: {from→dec_depth_from,
  to→dec_depth_to}}`, renderer получает готовое, не гадает;
- какой из двух — **определяет ПЕРВЫЙ реальный потребитель** (валидация
  «from < to», поиск пересечений интервалов, сравнение план-факт), а не
  сегодняшнее воображение. Все эти потребители сами в «Позже» (отчёты,
  валидация) — значит по строгому тесту Влада («без сущности НЕ обойтись?»)
  слой групп сегодня НЕ проходит: CRUD Листа2 собирается целиком без
  понятия «группа».

**Решение на СЕГОДНЯ: группы выводятся в подчинённые таблицы**, не через
тип, не через слой. Это безопасно для будущего: когда придёт сущность-
группа, подчинённые таблицы либо остаются, либо сворачиваются обратно
в поля — осознанная миграция под реального потребителя, не переделка
угаданного. Форма сущности-группы определяется потребителем — записано
здесь, чтобы будущая сессия НЕ прочитала «startdec отклонён» (он не
отклонён — отложен) и не потеряла идею.

**Уточнено повторно 2026-07-14, тем же тестом, до кода.** Вторая попытка
построить общий слой групп/весов (на этот раз — Claude, `model_registry`
kind=`group` + отдельная таблица весов на ПОЛЕ) остановлена Владом до
кода тем же вопросом «без сущности не обойтись?» — и найдена точная
причина, почему вес НЕ может жить на поле в принципе, не только сейчас:
поле — одно, а контекстов чтения, где у него есть вес/видимость/сосед по
группе, — много (сама таблица, и потом любая выборка/отчёт, объединяющий
несколько таблиц). Вес на поле отвечает на вопрос «чей это вес» только
пока потребитель один; со вторым отчётом вопрос обязателен, а ответа в
такой схеме нет — значит проектировать вес как атрибут поля было бы
готовить сущность под потребителя, чья форма ещё не решена, второй раз
подряд то же самое нарушение §15.8.

**Явно разделены два уровня, оба с прямым (не опосредованным) доступом
к данным, но с разным смыслом.** Базовый уровень — структура и
`dep_`/`rel_` (обслуживает CRUD, чинится инструментом ремонта БД,
07-11): видно всё, порядок — физический порядок колонок; своих данных
скрывать не от кого. Уровень представления — когда данные становятся
полезными для конкретной задачи чтения, не только существующими; вес,
видимость, группировка — свойства ЭТОЙ задачи, живут в определении
самого представления (отчёта/выборки), не в поле и не в реестре модели.
Отсюда следствие для будущего слоя отчётов, когда до него дойдёт
очередь: вес/видимость/группировка — часть состава конкретного отчёта
(«эти поля, в этом порядке, с такой группировкой»), не общая метаданная
модели. Единственный легитимный запрос к базовому уровню на сегодня —
не про группы вообще, а частный: у широких таблиц (`well`, 14 полей)
физический порядок колонок в БД иногда неудачен для чтения; чинится
прямой перестановкой колонок тем же инструментом ремонта структуры,
когда он до этого дорастёт — маленькая отдельная задача, не связанная
с отчётами и не срочная.

**`calc_` — отдельное сестринское семейство, НЕ подтип словаря**
(зафиксировано 2026-07-08, до реализации; имя уточнено 2026-07-14 —
дрейф к `math_` в одной из формулировок ниже был опережением, не
решением, снят). Общая философия резолвинга — «перейти по ссылке →
разрешить на границе сборки снапшота → подставить готовое значение» —
одна и та же для словарных проекций (а1/а2) и для вычислений: обе
следуют §7 (проверка на границе один раз) и §8 (рантайм не
интроспектирует). НО общий механизм ≠ общая таблица-описатель.
`voc_x` резолвит ИСТОЧНИК значения (откуда взять готовую строку);
`calc_x` резолвит СПОСОБ вычисления (как получить число из других
полей) — разные по форме паспорта, разные хендлеры, разные слоты
в снапшоте (`model.dictionaries` vs будущий `model.formulas`).
Слияние в одну таблицу нарушило бы §1 («имя поля выбирает сущность»
однозначно, без заглядывания внутрь паспорта) и §15.8 (абстракция
ради формы, не ради нужды) — риск назван прямо: рано или поздно
перестаёшь понимать по имени поля, что перед тобой. Спайк `ent_calc`
(entities.php, под карантинным баннером — см. журнал 2026-07-07/08)
концептуально доказал механику; следующий шаг (архивный план, п.14:
паспорт формулы → конфигуратор → родительская цепочка → кросс-табличное)
— первый в очереди сейчас, независимо от `ent_prm`/`ent_mlt` (те —
мёртвые легаси-форматы, не очередь, см. журнал 07-14).

**Гипотеза 07-08 (общий резолвер-движок со словарными проекциями)
проверена реальным проектированием 2026-07-14, подтверждена наполовину.**
Извлечение токенов `{поле}` из строки — переиспользуется буквально,
`template_parse()` без изменений годится и для формулы: тот же
синтаксис ссылки, та же грамотность на два места. РАЗРЕШЕНИЕ токена —
не переиспользуется и не должно: словарный шаблон склеивает токены в
строку, формула обязана вычислить арифметику (операторы, приоритет,
скобки) — заставить `template_parse` считать значило бы чинить
абстракцию под чужую задачу (§15.8), та же ошибка, что уже поймана
сегодня на группах. Формуле — свой маленький безопасный вычислитель
(разбор в дерево операций по белому списку, не `eval()`, тот же
принцип «текст не исполняется как код», что и в §16 п.3).

**Устройство паспорта (проектируется, не построено).** `model_formulas`
— своя таблица, не колонка в `model_labels` (критерий 07-11: формула
меняется другой рукой/ритмом, чем подпись): `dep_model_registry`
(строка реестра самого `calc_`-поля) + один столбец `data_formula` с
шаблоном `{поле} оператор {поле}` — токены и есть список переменных,
второго поля не нужно, так же как у `data_label_template`. Область
первого шага — только поля СВОЕЙ ЖЕ таблицы, реализуется структурно
(whitelist токенов = поля владеющей таблицы, не убеждением); шаг с
родительской цепочкой (архивный п.14, шаг 4) расширит whitelist,
код формулы переписывать не придётся.

**Найдено попутно, стоит закрепить — `calc_` и статистика по отчёту
различаются не сложностью формулы, а тем, сколько строк она видит.**
Влад предложил разделение «`calc_` для простого — `math_` про запас для
сложного»; при проверке тем же тестом (§15.8/«без сущности не обойтись»)
оказалось, что линия разреза не в сложности вычисления. `calc_`
(сколь угодно длинная формула, хоть с родительской цепочкой шага 4)
остаётся сущностью ОДНОЙ строки — обычное поле, обычный `field_exec`.
Статистика/агрегаты по ВЫБОРКЕ («средний расход по кусту», «процент
отклонений за квартал») не принадлежит ни одной записи вообще — это
свойство отчёта, не поля, ни при каком имени и ни при какой сложности
формулы. Значит `math_` как «сложный калькулятор про запас» не строится
никогда как отдельная сущность-сестра `calc_` — то, что под этим именем
имелось в виду, живёт в ещё не спроектированном слое отчётов, когда до
него дойдёт очередь. Перенесено в «Открытые вопросы» ниже, чтобы не
воспроизвестись повторно.

Состав слоя подтверждён референсной задачей; ничего не добавлять,
пока база не собирает тампонаж-CRUD.

## Открытые вопросы (зафиксированы, не проектируются)

- **статистика/агрегаты по выборке — не сущность поля ни при каком
  имени** (найдено 07-14, при проверке предложения «`calc_` для
  простого, `math_` про запас для сложного»): различие `calc_` и
  отчётной статистики не в сложности формулы, а в том, сколько строк
  она видит — одна (поле, `field_exec`) или выборка (отчёт). «Средний
  расход по кусту», «процент отклонений за квартал» не принадлежат ни
  одной записи — решаются вместе со слоем отчётов, не отдельной
  сущностью-сестрой `calc_`;
- портирование в `snapshot_init`: раздельная диагностика причин null,
  автоснятие мёртвого bootstrap-lock (>60 с), ожидание параллельной
  сборки, автосоздание `state/` — при первом касании snapshot-кода;
- **автоснятие осиротевшего `schema.lock`** (метка времени + `source`;
  протух → сирота) — переведено из «желательного» в **обязательное до
  расширения периметра писателей** (конфигуратор-как-писатель уже сейчас,
  позже — несколько инженеров): с N писателей вероятность падения под
  локом растёт линейно. Сегодня не блокер (ручной `rm state/schema.lock`),
  но предусловие мультиписательской фазы; стыкуется с аварийной сборкой
  класса А — сирота-лок растворяет ложный лок, аварийный режим подхватывает
  здоровую-но-нефайловую структуру, порознь каждый оставляет дыру;
- вес/видимость/группировка полей (план/факт, двухуровневые шапки) —
  атрибуты состава конкретного отчёта/выборки, не поля и не labels
  (уточнено 07-14, см. блок про группы выше); решается вместе с
  механизмом отчётов, не раньше;
- где живут правила валидации уровня модели (интервал ≤ глубины скважины,
  дочерняя запись не живёт без родительской, реагент принадлежит материалу);
- формат длительности «ч-мм» — при реализации ent_dur (composite-механика,
  единственный кандидат, у которого нет родного HTML-виджета);
- **`timestamp`** (аудит-штамп created_at/updated_at) — уточнено
  2026-07-05: «ноль кода» было неточно. MySQL проставляет ЗНАЧЕНИЕ сам
  (`DEFAULT CURRENT_TIMESTAMP`/`ON UPDATE`), но ИМЯ колонки всё равно
  обязано пройти `field_parse()` — свободного прохода не существует ни
  для чего (тот же класс, что вчера уронил `active`). Решение: новый
  структурный префикс (`sys_`?) в `STRUCTURAL_PREFIXES`, не строить,
  пока нет конкретного поля. Важно: имя `ent_time` уже занято другим
  смыслом (время суток, 2026-07-05) — для аудит-штампа нужно другое имя.

## Тестовая БД

Старая база под новым ядром не запускается. Legacy — источник предметки
(тампонаж: main, словари, зависимые таблицы), не архитектурного контракта.
Новая тестовая БД строится по правилам:

- `id` вместо `index`;
- никаких двоеточий в именах колонок;
- только поля с существующими сущностями;
- простые voc-таблицы как обычные таблицы;
- mch в будущем через junction table;
- model_registry + model_labels как служебные таблицы модели (subscr упразднена);
- root table — параметр модели, не hardcode `main`;
- данные из ТЗ-2 при переносе нормализуются (Excel-сериалы дат → date).

Подготовку дампа можно поручить ChatGPT на основе tt.sql и ТЗ-2.

# AI Context Protocol — спецификация компактных документов

Принято 2026-07-14 (журнал STATE.md). Цель — не ускорить чтение, а
избавить Клода от повторного восстановления состояния проекта в
начале каждой сессии. Наблюдение, на котором основано решение: одно
сообщение с архитектурным разбором (не код, не поиск — только
рассуждение) потребило 27% лимита сессии на свежем, только что
сброшенном лимите. Расход на «вспомнить, что уже понято» — реальная,
измеренная проблема, не теоретическая.

Это техзадание для Chat: какие документы вести, в какой форме, кто
их читает и кто пишет.

---

## 0. Общий принцип

Компактные документы — **навигационная карта**, не замена чтению
кода. Они говорят «что читать и где», не заменяют последний шаг —
сверку с живым кодом перед архитектурным решением. Механическая
работа по готовому шаблону может опираться на карту без этой сверки;
архитектурное решение — не может.

Язык: английский, короткие теги, без литературности, без разъяснений
для человека. Кириллица и «экзотические» кодировки не экономят —
токенизатор режет их менее эффективно, чем частую латиницу. Символы —
только частые, простые (✓ ✗ ⚠ → ←), не изобретённые пиктограммы:
редкий символ токенизируется дороже, чем короткий ASCII-тег.

## 1. FUNCTION_PASSPORTS_AI.md

Формат на функцию:

```
### <function_name>
sig:      <exact PHP signature>
loc:      <file>:<line_start>-<line_end>
commit:   <short SHA of last commit touching this range>
status:   ✓ | ⚠ | ✗

purpose:  <one line, dense>

calls:    <fn_a>, <fn_b>, ...
called_by:
  <file>:<line>  <caller>()  → <args, only if non-trivial>

effects:  [DB-R] [DB-W] [LOCK] [FILE] [EXIT] [PURE]
          (only tags that apply)

fail:     <null? [] ? exit? exception?>

contract: <caller invariants, only if non-obvious from signature>
```

`status`: ✓ — сверено с кодом на указанном commit. ⚠ — код после
этого commit менялся, паспорт не гарантированно точен. ✗ — известно
устарел, не использовать без чтения кода.

`effects` — фиксированный словарь, не расширять произвольно: `DB-R`
читает БД, `DB-W` пишет, `LOCK` берёт блокировку схемы, `FILE` пишет
на диск (снапшот), `EXIT` может прервать выполнение, `PURE` — ничего
из перечисленного.

`purpose` — одна плотная строка, не пересказ докблока абзацем: пересказ
теряет тонкости (пример дня: нюанс про `mysqli_insert_id` при UPDATE
против INSERT — abзацный парафраз такое сглаживает). Если деталь
важна — она должна остаться в самом коде/докблоке, паспорт лишь
показывает, что она может быть.

`called_by` — обязателен, не опционален: конкретное место и строка,
куда прыгнуть, не «функция где-то используется».

## 2. ARCHITECTURE_SNAPSHOT_AI.md

Структура — как в черновике Chat (§1 назначение, §2 формула исполнения,
§3 карта слоёв, §4 узлы, §7 запрещённые зависимости, §8 критерии
регрессии, §11 контекст по типу задачи) — эти секции уже доказали
пользу текстом, не менять произвольно.

Добавить обязательно:

```
generated_at_commit: <SHA>
sections_verified:   <list of §, if partial regeneration>
```

**Как это проверяется на стороне Клода (журнал 2026-07-14, важно для
Chat при обновлении):** `generated_at_commit` сверяется не со всем
репозиторием, а с файлами, которые ИМЕННО ЭТОТ документ описывает
(`git log <commit>..HEAD -- <scope>`). Для `ARCHITECTURE_SNAPSHOT_AI.md`
и `FUNCTION_PASSPORTS_AI.md` область — `*.php`. Правки `STATE.md`/
`HANDOFF.md`/документации НЕ делают эти два документа устаревшими.
Обновлять `generated_at_commit` в них нужно только после коммитов,
реально тронувших PHP-код, не после любого коммита в репозитории —
иначе документ будет казаться свежим, а генерироваться заново
избыточно часто без причины.

Каждая секция, перечисляющая конкретные функции/файлы (§4, §6, §11),
несёт те же `loc`/`commit`-метки, что и паспорта — не голый текст без
привязки к версии кода.

Убрать: развёрнутые ASCII-диаграммы стрелочек (карта слоёв) — не
доказано, что дешевле текстового списка того же смысла (§7 уже даёт
то же самое текстом). Разрешить оставить, если реально экономит по
факту, не по ощущению — но не обязательный элемент.

## 3. STATE_INDEX_AI.md

Не заменяет `STATE.md` (тот остаётся прозой, читает человек). Индекс
— только для быстрого «искали мы это раньше?», до похода в сам журнал.
Раздел «Сейчас» (текущая задача) читается напрямую всегда, минуя
индекс — он слишком мал и слишком важен, чтобы через что-то проходить.

**Область сверки `generated_at_commit`:** `STATE.md` конкретно
(`git log <commit>..HEAD -- STATE.md`), не весь репозиторий и не
`*.php` — правки кода не делают этот индекс устаревшим, только правки
`STATE.md` делают.

```
date | tags | decision (one line) | ref (file:lines)
```

Пример:
```
07-13 | link_, dict | Idea A shipped, Idea B deferred | STATE.md:473-490
```

## 4. HANDOFF (формат отчёта Клода после сессии/крупного куска)

```
## HANDOFF

task: <short title>

decision: <what was decided architecturally, if anything>

changed:
  - file::function
added:
  - file::function
removed:
  - file::function

contracts_changed:
  - <contract name>: old → new

contracts_unchanged:
  - <contract> [checked: read | grep | none]

docs_to_update:
  - function_passports: <list>
  - architecture_snapshot: <§>
  - state_index

delegate_tasks:
  - file: <path>
    operation: <what>
    constraints:
      - <invariant to preserve>

verification:
  - php -l ...
  - smoke: <pass/fail, which sections>
```

**`contracts_unchanged` — самое ценное и самое опасное поле одновременно.**
Ценное — потому что разрешает НЕ открывать файл вообще, не просто
экономит строки чтения. Опасное — потому что это утверждение об
ОТСУТСТВИИ изменения, а доказать отсутствие труднее, чем перечислить
присутствие: небрежное «наверное, не трогал» здесь опаснее пропуска
в `changed`. Обязательная пометка способа проверки:
- `[checked: read]` — файл открыт и прочитан целиком/фрагментом.
- `[checked: grep]` — только проверено отсутствие упоминаний по
  паттерну (слабее, годится для узких утверждений).
- `[checked: none]` — предположение по памяти, не проверено явно —
  использовать редко, только когда уверенность действительно высокая
  (например, файл не открывался в этой сессии вообще).

Периодическая сверка доверия: раз в несколько сессий (не каждый раз)
— взять случайный паспорт/утверждение `contracts_unchanged` из
накопленного и проверить против живого кода. Не автоматизировать
формально сейчас — просто не забывать это делать, как страховку от
незаметного расхождения цепочки с реальностью.

## 5. Что безопасно делегировать без полного контекста

Задачи с уже определённым контрактом: добавить паспорт по шаблону,
обновить `called_by`, заменить вызов одной функции на другую по
точной инструкции, привести сигнатуры к шаблону, обновить
`STATE_INDEX`, проверить ссылки на удалённые функции, сравнить
паспорт с изменившимся файлом, механический перенос по образцу
(пример дня — `mysqli_*` → `db_select`/`db_execute`, шаблон см. ниже).

## 6. Что нельзя делегировать без полного архитектурного контекста

Изменение смысла сущности; новый тип связи; изменение границы
ядро/сущность; изменение структуры снапшота; новый механизм запросов;
выбор между конвенцией имени и системной записью; изменение доверенной
границы request; что считается моделью, а что кодом. Решает Клод с
Владом, реализация после решения — делится свободно.

## 7. Образец промпта для механического переноса (реальный, не абстрактный)

```
Task: move direct mysqli_* calls in labels.php/helpers.php/index.php
to db_select()/db_execute() from db.php — same two functions already
used in core.php and configurator.php (use those as reference pattern).

Rules:
1. SELECT with params → db_select($db, $sql, $types, $params).
2. SELECT without params (trusted table/field name interpolated
   directly into text) → db_select($db, $sql), no types.
3. INSERT/UPDATE/DELETE → db_execute(), same params principle.
4. INSERT followed by mysqli_insert_id() → take 'id' from db_execute
   response instead of separate call.
5. DO NOT touch: mysqli_connect/mysqli_close/mysqli_set_charset
   (connection management, not a query — out of wrapper scope).
6. DO NOT change SQL text itself — only execution method.
7. After each function: check whether failure-path behavior changed
   (silent [] vs explicit early return etc.) — if a difference
   exists, comment it explicitly, do not stay silent.

Output as unified diff per file, brief comment under each spot —
what moved, whether behavior differs.
```

## 8. Рабочий цикл

```
1. Влад + Клод принимают архитектурное решение.
2. Клод меняет код, проверяет (php -l, smoke).
3. Клод выдаёт HANDOFF (§4 выше).
4. Chat: обновляет паспорта, snapshot, STATE_INDEX, выполняет
   делегированные локальные правки (§5).
5. Gemini (если подключён): массовые шаблонные операции.
6. Следующая сессия Клода получает: ARCHITECTURE_SNAPSHOT_AI.md +
   STATE_INDEX_AI.md + последний HANDOFF + паспорта только затронутых
   функций + сама задача. Не весь STATE.md, не весь ARCHITECTURE.md,
   не все паспорта, не всё ядро.
```

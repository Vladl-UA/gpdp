# GPDP/RNA — Онбординг ИИ (HANDOFF)

Первый файл, который читает ИИ, продолжающий разработку. Не повторяет правила
(они в ARCHITECTURE.md) и не повторяет состояние (оно в STATE.md). Отвечает на
три вопроса: **что читать, в каком порядке и с какой ролью; чему верить при
расхождении; чего не делать.** Цель — минимальная потеря контекста при смене
сессии или модели.

Меняется редко: при изменении состава носителей контекста или протокола работы,
не при написании кода.

---

## 0. Первая строка сессии

Если пришла автоматически (project custom instructions — см. §8) — эта
строка не нужна отдельно, протокол §8 запускается сам. Если нет —
первое сообщение человека:

```
Работаем по ARCHITECTURE.md. Текущая задача — пункт N из раздела «Сейчас» STATE.md.
```

Пересказ прошлой сессии не требуется и не приветствуется: история диалогов — не
источник истины (§2). Всё нужное лежит в файлах репозитория.

**Первое действие ИИ, до первого содержательного ответа:**

1. Прочитать `STATE.md`, раздел «Сейчас» — напрямую, целиком, не через
   AI-документы никогда: он маленький, это сама постановка задачи,
   и он самый свежий по определению (правится в начале и в конце
   каждой сессии). AI-документы не заменяют этот шаг, они идут после.

2. Задача известна — проверить каждый из трёх AI-документов **отдельно**.
   Точка отсчёта — не поле `generated_at_commit` внутри документа
   (самоотчёт, может быть забыт/сбит), а реальный git-коммит,
   последним тронувший сам файл документа — берётся механически,
   не зависит ни от чьей памяти (уточнение 2026-07-14, предложение
   Влада — надёжнее вчерашней версии на самоотчётном поле):

   ```
   LAST=$(git log -1 --format=%H -- <AI-документ>)
   git log $LAST..HEAD --oneline -- <область>
   ```

   Область для каждого документа — файлы, которые он реально описывает
   (журнал 2026-07-14: широкая проверка по всему репозиторию давала
   ложную тревогу — правка `STATE.md` показывала «устарело» даже
   документы про код, которые код не трогал вовсе):

   ```
   FUNCTION_PASSPORTS_AI.md / ARCHITECTURE_SNAPSHOT_AI.md (описывают код):
     область — '*.php'

   STATE_INDEX_AI.md (описывает журнал STATE.md):
     область — STATE.md
   ```

   Пусто для конкретного документа — он свежий, читать вместо
   соответствующей тяжёлой части (снапшот вместо `ARCHITECTURE.md`
   целиком, паспорта вместо чтения кода вслепую, индекс вместо журнала
   `STATE.md` целиком). Не пусто — этот документ устарел, читать
   первоисточник как обычно; каждый документ проверяется и
   используется независимо от других, один устаревший не тянет за
   собой остальные.

3. По снапшоту/паспортам — построить логику: что нужно менять и где.
   **Это не отменяет чтение кода** — паспорта дают адрес (`file:line`,
   `called_by`), сам код по этому адресу открывается и читается перед
   правкой как обычно (§15, «архитектура до кода» — не отменяется
   компактными документами, они экономят ПОИСК нужного места, не
   заменяют чтение найденного).

**Дешёвая проверка свежести — не то же самое, что содержательная
сверка.** Шаги 2 выше — это несколько коротких команд `git log`, без
диффов и без чтения содержимого; стоимость пренебрежимо мала. Не
путать с диффами/grep/построчной проверкой утверждений против кода —
та была разовой оценкой качества первой версии документов 2026-07-14
(STATE.md, журнал), не повторяющимся шагом протокола, и делается
редко, вручную, по отдельному запросу.

## 2. Иерархия источников истины

```
«как должно быть»       → ARCHITECTURE.md
«как есть на самом деле» → код + smoke_test.php
«где мы и почему так»    → STATE.md (состояние + журнал)
история диалогов         → НЕ источник; пересказы не переносятся
```

Расхождение кода с ARCHITECTURE.md — это баг для доклада, не тихое правило к
исполнению. Один источник истины на факт; дубль состояния — ошибка (§12).

**Project knowledge (файлы, прикреплённые к Claude-проекту) — не отдельный
уровень иерархии, а ручной кэш GitHub.** Обновляется заливкой человеком,
без сигнала об отставании. 2026-07-10 — не гипотеза: `index.php` в
project knowledge был черновиком до пайплайна, `STATE.md` — младше
реальности на 18 записей журнала, и ни один инструмент не предупредил
об этом сам. При сомнении — сверяться с GitHub `main` (§8), не с тем,
что «видно» в project knowledge.

## 3. Чего не делать (передача дисциплины)

Главный риск смены ИИ — не потеря фактов, а дрейф дисциплины: соблазн «улучшить»
архитектуру, добавить оборону, переобъяснить. Прямые запреты:

- **Не релитигировать журнал.** Отклонённое отклонено с причиной. Если кажется,
  что решение надо вернуть, — это новое решение с записью в журнал, не тихое
  «исправление».
- **Не писать оборонительный код против несуществующих угроз** (§13). Среда
  закрытая и контролируемая; проверки живут на границе, один раз.
- **Не перепроверять внутри трубы** (§7). Пути мимо границы не существует по
  построению — дубль проверки лишь размывает источник истины.
- **Не вносить предметную модель в ядро**, включая пустые крючки «на будущее»
  (§12). Пустое место в коде — уже утверждение о модели.
- **Не плодить абстракцию** там, где хватает
  `($passport['handlers'][$mode])($data, $mode)` (§15.8).
- **Эпистемическая гигиена.** Различать «спроектировано верно» и «проверено на
  целевом localhost». Не overclaim. Журнал не утверждает больше проверенного на
  месте — держать эту планку при любом изложении проекта, внутреннем и внешнем.

## 4. Проверка любого предложения

- **Контрольные вопросы — §15** ARCHITECTURE.md. Пройти явно, по номеру пункта,
  не «общим ощущением согласованности с духом».
- **Анти-паттерны — §12.** Сверить перед тем, как показать решение.
- **Осторожно, устаревшая ссылка.** Инструкции проекта пока говорят «контрольные
  вопросы §12». После вставки разделов о пространстве имён и тонкой трубе это
  стало **§15**; §12 — анти-паттерны. Одна цифра, ещё не поправленная в
  инструкциях: сверяться по §15, не по §12.

## 5. Роли и язык

- **Архитектура и планирование** — Opus-класс.
- **Имплементация** — Fable; начинается только по явному запросу смены модели.
- **ChatGPT / Gemini** — консультанты; их выводы входят в проект только через
  Vlad, не напрямую в файлы.
- **Язык:** обсуждение и документы — русский; код и идентификаторы — английский.

## 6. Критерий «шаг завершён»

1. Смоук зелёный на **целевом localhost** (не только в среде разработки).
2. Решение записано в журнал STATE.md с датой; раздел «Сейчас» обновлён.
3. В снапшоте нет факта, которого нет в DDL / реестре / паспортах (§8):
   удаление снапшота + запрос даёт эквивалентную пересборку.

## 7. Грабли развёртывания (симптом → где смотреть в STATE.md)

- Система отвечает 503 «вечно» → осиротевший `state/schema.lock` (`finally` не
  срабатывает при fatal/exit). Лечение: `rm state/schema.lock`. Журнал 2026-07-03.
- «Мигающий» EROFS, сайт лежит при здоровой БД → `ProtectHome=read-only`
  веб-службы; drop-in `ReadWritePaths=<путь>/state`. Журнал 2026-07-04.
- Смоук валит секции 1/6 на чистом коде → разошёлся эталонный дамп. Новый
  localhost = залить эталонный дамп + прогнать смоук. Раздел «Тестовая БД».
- Проверка смоука падает `Permission denied` в подпроцессе → смоук живёт в одном
  процессе, без `PHP_BINARY`/subprocess. Журнал 2026-07-08.

## 8. Живая синхронизация — GitHub как источник истины (с 2026-07-10)

Репозиторий: `github.com/Vladl-UA/gpdp`, публичный, ветка `main`.
Легаси-код (дочищенный 2026-07-10) — в ветке `legacy`, архив, не трогать.

**Начало сессии, до любого предложения по существу** — подтянуть три
файла напрямую с GitHub, не из project knowledge (bash, репозиторий
публичный, токен для чтения не нужен):

```bash
curl -s https://raw.githubusercontent.com/Vladl-UA/gpdp/main/HANDOFF.md
curl -s https://raw.githubusercontent.com/Vladl-UA/gpdp/main/STATE.md
curl -s https://raw.githubusercontent.com/Vladl-UA/gpdp/main/ARCHITECTURE.md
```

Если ответ расходится с project knowledge — верить GitHub, явно доложить
расхождение, не молчать и не гадать, какой источник новее.

**Объективная проверка живого кода**, отдельно от чтения документов: при
касании кода — попросить Влада прогнать `smoke_test.php` на целевом
localhost. Документ может быть свежим, а реально задеплоенный на сервере
код — ещё не догнал коммит; это две разные вещи, обе стоит проверить.

**Конец сессии**, если были правки кода/STATE.md/HANDOFF.md: закоммитить
и запушить в `main` ДО завершения сессии, не «сделаю потом» — эта
формулировка и породила инцидент 2026-07-10. Отдать Владу файл(ы) для
деплоя тем же сообщением, где коммит.

**Токен GitHub не переживает сессию** (контейнер сбрасывается между
диалогами) — Влад вставляет fine-grained PAT (права `Contents` +
`Pull requests`, только репозиторий `gpdp`) заново в начале любой сессии,
которая должна писать в репозиторий. Сессии, которые только читают
(§ выше), токен не нужен вовсе — репозиторий публичный.

**Однострочный якорь версии** в код-файлах (не в md — они самодатируются
журналом STATE.md): `// sync: ДАТА, STATE.md пункт N (кратко)` —
перезаписывается при каждой правке, не растёт построчно. Не хеш коммита
(циклическая зависимость — файл не может содержать хеш коммита, в
который сам войдёт); дата+тема достаточно, чтобы найти нужный коммит в
истории GitHub при необходимости.

**Автоматизация начала сессии** (опционально, но закрывает разрыв
полностью): в Claude Project → Settings → Custom instructions — текст,
который заставляет протокол выше выполняться без первой строки от
человека. Хранится в самих настройках Claude, не в файлах репозитория
(это не тот же механизм, что project knowledge, и не подвержен той же
проблеме отставания — инструкция статична, не файл для перезаливки).

**Флаг пересборки снапшота — `SNAPSHOT_REBUILD_REQUIRED`** (журнал
2026-07-13). Снапшот на диске (`state/`) не знает о правках КОДА
резолвера сам по себе — обновляется только как побочный эффект
структурных действий конфигуратора. Правка `core.php`/`helpers.php`/
`entities.php`, меняющая `field_parse`/`snapshot_build_*`/`lookup_*`
(не каждая правка подряд — `render.php`/документация не требуют),
**обязана** сопровождаться созданием пустого файла
`SNAPSHOT_REBUILD_REQUIRED` в том же коммите. Git-хук `post-merge`
(`.githooks/post-merge`, версионируется — `git config core.hooksPath
.githooks`, настраивается один раз) после каждого `git pull` сам видит
флаг, вызывает `tools_rebuild_snapshot.php`, убирает флаг. Забыть флаг
при системной правке — то же самое, что вчерашняя забытая инвалидация
OPcache: код новый, данные старые, тихое расхождение вместо явной
ошибки.

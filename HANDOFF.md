# GPDP/RNA — Онбординг ИИ (HANDOFF)

Первый файл, который читает ИИ, продолжающий разработку. Не повторяет правила
(они в ARCHITECTURE.md) и не повторяет состояние (оно в STATE.md). Отвечает на
три вопроса: **что читать, в каком порядке и с какой ролью; чему верить при
расхождении; чего не делать.** Цель — минимальная потеря контекста при смене
сессии или модели.

Меняется редко: при изменении состава носителей контекста или протокола работы,
не при написании кода.

---

## 0. Первая строка сессии

```
Работаем по ARCHITECTURE.md. Текущая задача — пункт N из раздела «Сейчас» STATE.md.
```

Пересказ прошлой сессии не требуется и не приветствуется: история диалогов — не
источник истины (§2). Всё нужное лежит в файлах репозитория.

## 2. Иерархия источников истины

```
«как должно быть»       → ARCHITECTURE.md
«как есть на самом деле» → код + smoke_test.php
«где мы и почему так»    → STATE.md (состояние + журнал)
история диалогов         → НЕ источник; пересказы не переносятся
```

Расхождение кода с ARCHITECTURE.md — это баг для доклада, не тихое правило к
исполнению. Один источник истины на факт; дубль состояния — ошибка (§12).

## 3. Чего не делать (передача дисциплины)

Главный риск смены ИИ — не потеря фактов, а дрейф дисциплины: соблазн «улучшить»
архитектуру, добавить оборону, переобъяснить. Прямые запреты:

- **Не релитигировать журнал.** Отклонённое отклонено с причиной. Если кажется,
  что решение надо вернуть, — это новое решение с записью в журнал, не тихое
  «исправление».
- **Не писать оборонительный код против несуществующих угроз** (§13). Среда
  закрытая и контролируемая; проверки живут на границе, один раз.
- **Не перепроверять внутри трубы** (§7). Пути мимо границы не существует по
  построению — дубль проверки лишь размывает источник истины.
- **Не вносить предметную модель в ядро**, включая пустые крючки «на будущее»
  (§12). Пустое место в коде — уже утверждение о модели.
- **Не плодить абстракцию** там, где хватает
  `($passport['handlers'][$mode])($data, $mode)` (§15.8).
- **Эпистемическая гигиена.** Различать «спроектировано верно» и «проверено на
  целевом localhost». Не overclaim. Журнал не утверждает больше проверенного на
  месте — держать эту планку при любом изложении проекта, внутреннем и внешнем.

## 4. Проверка любого предложения

- **Контрольные вопросы — §15** ARCHITECTURE.md. Пройти явно, по номеру пункта,
  не «общим ощущением согласованности с духом».
- **Анти-паттерны — §12.** Сверить перед тем, как показать решение.
- **Осторожно, устаревшая ссылка.** Инструкции проекта пока говорят «контрольные
  вопросы §12». После вставки разделов о пространстве имён и тонкой трубе это
  стало **§15**; §12 — анти-паттерны. Одна цифра, ещё не поправленная в
  инструкциях: сверяться по §15, не по §12.

## 5. Роли и язык

- **Архитектура и планирование** — Opus-класс.
- **Имплементация** — Fable; начинается только по явному запросу смены модели.
- **ChatGPT / Gemini** — консультанты; их выводы входят в проект только через
  Vlad, не напрямую в файлы.
- **Язык:** обсуждение и документы — русский; код и идентификаторы — английский.

## 6. Критерий «шаг завершён»

1. Смоук зелёный на **целевом localhost** (не только в среде разработки).
2. Решение записано в журнал STATE.md с датой; раздел «Сейчас» обновлён.
3. В снапшоте нет факта, которого нет в DDL / реестре / паспортах (§8):
   удаление снапшота + запрос даёт эквивалентную пересборку.

## 7. Грабли развёртывания (симптом → где смотреть в STATE.md)

- Система отвечает 503 «вечно» → осиротевший `state/schema.lock` (`finally` не
  срабатывает при fatal/exit). Лечение: `rm state/schema.lock`. Журнал 2026-07-03.
- «Мигающий» EROFS, сайт лежит при здоровой БД → `ProtectHome=read-only`
  веб-службы; drop-in `ReadWritePaths=<путь>/state`. Журнал 2026-07-04.
- Смоук валит секции 1/6 на чистом коде → разошёлся эталонный дамп. Новый
  localhost = залить эталонный дамп + прогнать смоук. Раздел «Тестовая БД».
- Проверка смоука падает `Permission denied` в подпроцессе → смоук живёт в одном
  процессе, без `PHP_BINARY`/subprocess. Журнал 2026-07-08.

# Формат JSON для оптовой загрузки (tools_bulk_import.php)

Справочный документ для генерации файлов заливки (журнал 2026-07-15,
шаг 2/5). Не техническая документация конвейера — практическая шпаргалка
для составления конкретных файлов данных.

## Структура верхнего уровня

```json
{
  "имя_таблицы": [ {запись}, {запись}, ... ],
  "другая_таблица": [ {запись}, ... ]
}
```

Ключ — имя таблицы **как в базе** (`example_person`, `well`), не
человеческое название. Значение — массив записей этой таблицы.
Порядок ключей верхнего уровня не важен — каждая таблица обрабатывается
независимо, если только одна из них не вложена в другую (см. ниже).

## Одна запись

```json
{"поле": "значение", "поле2": "значение2", "дочерняя_таблица": [{...}]}
```

**Обычные поля** (`data_`, `int_`, `dec_`, `date_`, `bul_`, ...) —
значение как строка, даже для чисел и дат: `"12"`, `"2026-07-15"`,
`"1"`/`"0"` для `bul_`. `record_save()` сам проверяет и приводит тип по
паспорту сущности — файл заливки не обязан знать, что `dec_` хочет
именно число, а `bul_` — именно 0/1.

**Словарные поля** (`voc_`, `link_`) — **человеческое название**, не
внутренний id: `"voc_smoke": "Приобское"`, не `"voc_smoke": "3"`.
Разрешается в id автоматически (`lookup_id_by_label`, точное
совпадение, при неоднозначности — явный отказ этой записи, не угадывание).

**Структурные поля** (`id`, `rel_main`, `active`, `dep_*`) — **не
указываются в файле вообще**. `id` генерируется базой сама. `dep_*`
подставляется автоматически из вложенности (см. ниже) — писать его в
JSON не нужно и бессмысленно, он будет отвергнут как неизвестное поле
(структурные поля в v1 загрузчика намеренно не читаются из файла —
доверенный канал, не request, см. `tools_bulk_import.php`, докблок).
`rel_main` явно **не поддерживается** в v1 — только `dep_`-дерево.

## Дерево (родитель → дети)

Дочерняя запись — вложенный массив **под именем таблицы-ребёнка**,
внутри объекта родителя:

```json
{
  "well": [
    {
      "data_name": "Скважина №2",
      "voc_mr": "Приобское",
      "cementing_stage": [
        {
          "int_stage_number": "2",
          "dec_pressure_work": "14",
          "dec_pressure_stop": "18",
          "cementing_interval": [
            {
              "dec_depth_from": "2000",
              "dec_depth_to": "1000",
              "voc_material": "ОТМ-5",
              "dec_amount": "20",
              "cementing_buffer": [
                {"dec_dec_volume": "3", "voc_composition": "МБП-М", "dec_weight": "120"}
              ]
            }
          ],
          "cementing_reper": [
            {
              "dec_volume": "3", "dec_density": "1.48",
              "cementing_reper_component": [
                {"voc_component": "ПЦТ-G-CC-1"},
                {"voc_component": "КМЦ", "dec_amount": "0.01"}
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

Ключ распознаётся как «дочерняя таблица», если такая таблица реально
существует в структуре (не по префиксу имени ключа, не по совпадению
с чем-то похожим — по факту). Порядок вставки — сверху вниз, родитель
раньше детей: `dep_*`-значение подставляется из `id` только что
вставленного родителя, доступного к этому моменту. Один родитель на
таблицу — конвейер `dep_` не поддерживает множественных родителей,
второй уровень вложенности с тем же именем таблицы не нужен и не
предусмотрен.

## Явные отказы (не молчание)

- Неизвестное поле или ключ, не совпадающий ни с полем, ни с дочерней
  таблицей — отказ этой записи, с именем ключа в сообщении.
- Значение словарного поля не найдено в справочнике — отказ, с точным
  текстом, что искали.
- Значение словарного поля неоднозначно (два id — одна подпись) —
  отказ, с перечнем всех совпавших id.
- Отказ на любом уровне — дети этой записи не обрабатываются вовсе
  (родителя нет, вставлять некуда).
- Отчёт построчный, до конца файла — не останавливается на первой
  ошибке (важно видеть все проблемы разом, не по одной за прогон).

## Что проверить перед реальной вставкой

Всегда сначала `--dry-run` — та же логика, реальная транзакция с
откатом, ничего не остаётся в базе, кроме — честно — нескольких
пропущенных номеров `id` у следующей настоящей вставки (`AUTO_INCREMENT`
у InnoDB не откатывается транзакцией).

## Пример плоского случая (без дерева)

```json
{
  "example_person": [
    {"data_name": "Иван Петров", "link_favorite_color": "Синий", "link_unloved_color": "Оранжевый"},
    {"data_name": "Мария Соколова", "link_favorite_color": "Зелёный", "link_unloved_color": "Красный"}
  ]
}
```

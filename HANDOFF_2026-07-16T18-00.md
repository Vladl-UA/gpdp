## HANDOFF

task: Postgres migration, шаги 1-2 (STATE.md «Сейчас» п.9) — связь/подключение и типы сущностей

decision: Полный катор MySQL → Postgres, не dual-driver (решение Влада 07-16).
  db.php остаётся единственным местом с диалектной логикой (?→$N
  конвертация плейсхолдеров, RETURNING-парсинг id) — вызывающий код
  (record_*/configurator_*/lookup_*) не тронут по текстам SQL, только
  по типу параметра соединения. Подробности и найденные по ходу баги
  (model_registry.active boolean/integer, ent_bul tinyint(1)) — STATE.md
  «Сейчас» п.9, шаги 1-2, с полным разбором.

changed:
  - db.php::db_select — mysqli_query/mysqli_prepare → pg_query/pg_query_params;
    $types теперь означает только "есть параметры" (Postgres не требует
    типовую строку), не отброшен из сигнатуры ради нулевой правки вызывающих
  - db.php::db_execute — то же + новая механика: 'id' в результате заполняется
    ТОЛЬКО если сам текст SQL содержит `RETURNING` (регистронезависимо);
    db.php больше не читает last_insert_id неявно — Postgres не имеет аналога
  - core.php::snapshot_build_structure — SHOW TABLES/SHOW COLUMNS →
    information_schema.columns + pg_catalog.pg_tables + отдельный запрос
    первичного ключа (pg_index/pg_attribute); формат возврата полей
    (name/kind/entity/db_type/nullable/key) не изменился
  - core.php::record_save — обратные кавычки убраны из динамического SQL;
    INSERT-ветка получила `RETURNING id` (раньше id брался из
    mysqli_insert_id неявно)
  - configurator.php::configurator_create_table — DDL-билдер: id-колонка
    `GENERATED ALWAYS AS IDENTITY` вместо `AUTO_INCREMENT`, структурные
    dep_/rel_main колонки `INTEGER` вместо `INT UNSIGNED`, `ENGINE=/
    CHARSET=/COLLATE=` убраны целиком (не табличный уровень в Postgres);
    два INSERT в model_registry получили `RETURNING id` (их id читается
    сразу после, раньше — mysqli_insert_id)
  - configurator.php::configurator_add_field, configurator_adopt_field,
    configurator_adopt_table — INSERT в model_registry получил `RETURNING id`
    (тот же класс правки, что в create_table)
  - labels.php::labels_save_label — `ON DUPLICATE KEY UPDATE ... VALUES(x)`
    → `ON CONFLICT (dep_model_registry) DO UPDATE SET x = EXCLUDED.x`
  - helpers.php::admin_db_connect — тело переписано на db_connect() +
    try/catch → http_response_code(500); возвращаемый тип PgSql\Connection
  - tools_bulk_import.php (dry-run транзакция) — mysqli_begin_transaction()/
    mysqli_rollback() → pg_query($db, 'BEGIN')/pg_query($db, 'ROLLBACK')
    (эта функция ДАЖЕ НЕ ИМЕЕТ записи в текущих паспортах — весь файл
    tools_bulk_import.php описан только пятью функциями, транзакционный
    блок в конце файла — top-level код, не отдельная функция, паспорту
    подлежит не он, а корректность purpose/effects функций, которые он
    вызывает, если они не изменились — см. contracts_unchanged)
  - entities.php::ent_bul — db.default_type 'tinyint(1)' → 'smallint'
    (Postgres не имеет tinyint; boolean сознательно НЕ выбран — см.
    комментарий в самом entities.php и STATE.md «Сейчас» п.9 шаг 2)

added:
  - db.php::db_connect(array $cfg): PgSql\Connection — единая точка
    подключения, заменяет 6 мест дублирования (helpers.php::admin_db_connect
    + четыре консольных скрипта: diag_refresh.php, smoke_test.php,
    tools_bulk_import.php, tools_rebuild_snapshot.php). Бросает
    RuntimeException при отказе (не exit()) — вызывающий код сам решает
    HTTP 500 против текста в stdout
  - db.php::db_close(PgSql\Connection $db_connection): void
  - db.php::db_placeholders(string $sql): string — внутренний конвертер
    '?' → '$1,$2,...', вызывается изнутри db_select/db_execute, наружу
    не экспортируется как публичный контракт (но физически вызываема)

removed:
  - (ничего не удалено функционально; helpers.php::admin_db_connect тело
    заменено, не удалено — сигнатура прежняя, только возвращаемый тип)

contracts_changed:
  - db_select/db_execute: параметр $types раньше был буквальным mysqli
    type-string ('s','i','d'...) для mysqli_stmt_bind_param; теперь
    любое НЕПУСТОЕ значение просто означает "есть параметры,
    используй pg_query_params" — конкретные буквы больше не имеют
    смысла, но вызывающий код (везде передаёт настоящие mysqli-строки
    вроде 'iss') не пришлось трогать, потому что параметр только
    проверяется на пустоту/непустоту, содержимое игнорируется
  - db_execute: поле 'id' в результате раньше = mysqli_insert_id() всегда
    (для не-INSERT операций — мусорное значение, ответственность
    вызывающего кода не читать его); теперь = 0 если в SQL нет
    RETURNING, иначе — значение из RETURNING-строки. Семантически
    строже (нет мусора вместо явного нуля), но код, полагавшийся на
    "любой ненулевой mysqli_insert_id даже без явного расчёта на него",
    если такой был — с этим стоит свериться отдельно (я не нашёл ни
    одного вызова db_execute, читающего ['id'] без RETURNING в SQL —
    но искал grep'ом, не построчным аудитом каждого вызова)

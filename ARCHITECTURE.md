# GPDP/RNA — Архитектурная конституция

Статус: **замороженные решения**. Документ меняется только при пересмотре решения, не при написании кода.
Философская мотивировка решений — в GPDP-RNA_Architecture.pdf (том 2). Доктрина словарей и классификации связей — в О_словарях.md. Здесь — только директивы.

---

## 1. Главный принцип

GPDP — универсальная платформа. Приложение = конфигурация GPDP, записанная в БД.

```
БД хранит модель.
Имя поля выбирает сущность.
Паспорт сущности выбирает handler.
Handler получает данные и режим.
Handler возвращает структурированный результат.
Renderer превращает результат в нужный формат.
```

**Тест для любого кода ядра:** можно ли подключить этот код к чужой БД с другой
предметной моделью без единой правки? Да → код принадлежит ядру.
Нет → это модельная информация (её место в БД) или ошибка проектирования.

Код не знает ничего про модель данных. Табу: предметная область в коде,
включая «пустые крючки на будущее» — пустое место в коде уже является
утверждением о модели.

## 2. Центральная операция и цепочка доверия

```php
$result = ($passport['handlers'][$mode])($data, $mode);
```

Цепочка доверия (нарушение любого звена — блокер):

```
untrusted request
  → action из whitelist → action_map → внутренний $mode
  → table/field проверены по schema snapshot
  → field_parse() → сущность
  → паспорт из entity registry (доверенный)
  → callable ТОЛЬКО из паспорта
  → вызов
```

Инварианты:
- request никогда не содержит имён вызываемых функций;
- пользователь выбирает действие, ядро выбирает режим, паспорт выбирает функцию;
- hidden-поля формы — подсказки, не факты; каждая проверяется по модели;
- обычный route не имеет пути к schema/admin-функциям, у admin-контура
  отдельный dispatcher, отдельные проверки прав и lock-механизм.

## 3. Контракт handler'а

**Сигнатура (единая для всех сущностей и режимов):**

```php
function xxx_handler(array $data, string $mode): array
```

**Рабочие режимы сущности (канонический словарь):** `new`, `edit`, `read`, `validate`.
Имена `form_new`/`form_edit` упразднены.

**Режима `save` у сущности нет.** Физическую запись (INSERT/UPDATE, prepared
statements) выполняет универсальный pipeline после validate всех полей.
Сущность проверяет и нормализует; pipeline пишет. Ordinary pipeline имеет
жёсткий whitelist рабочих режимов и физически не способен вызвать
`create`/admin-режимы, даже если они есть в паспорте.

- Ядро выбирает handler по ключу `$mode` в `handlers` паспорта.
- Один handler на несколько режимов — законно; решается паспортом.
- Handler доверяет аргументу `$mode`, не угадывает режим из содержимого `$data`
  (один источник истины на вызов).
- Handler'ы могут делегировать общим helper'ам
  (`lookup_options` из helpers.php, ...) — это норма.

**`$data` — подготовленный пакет, никогда не сырой request.**
Подготовку выполняет pipeline до вызова. Минимальный состав:

```php
[
    'table' => 'main',
    'field' => [
        'raw'    => 'data_title',
        'entity' => 'data',
        'name'   => 'title',
        'labels' => [...],  // строка labels КАК ЕСТЬ; нужную колонку выбирает потребитель
        'schema' => [ 'db_type' => 'varchar(255)', 'nullable' => true, 'key' => '' ],
    ],
    'value' => null,   // текущее/нормализованное значение (по режиму)
    'row'   => null,   // строка записи, если режим её требует
]
```

**Контракт режима `validate`:**

```php
[ 'valid' => true,  'value' => $normalized, 'errors' => [] ]
[ 'valid' => false, 'value' => null,        'errors' => ['Поле обязательно'] ]
```

Сущность со сложным хранением (m2m, file) в будущем возвращает не прямое
значение, а storage plan (`'storage' => ['type' => 'm2m', ...]`) — запись
всё равно выполняет pipeline, сущность SQL не пишет.

**Контракт результата операции записи (возвращает pipeline, честно,
по стандартам работы с БД):**

```php
[ 'ok' => true, 'operation' => 'insert'|'update', 'table' => 'main',
  'id' => 123, 'affected_rows' => 1, 'errors' => [] ]
```

**Возврат — структурированный массив, не HTML.** HTML — лишь один из renderer'ов
(наравне с JSON, CLI, отчётами). Базовая форма результата:

```php
[ 'type' => 'input', 'widget' => 'text', 'name' => 'data_title',
  'label' => 'Назва', 'value' => '', 'meta' => [...] ]

[ 'type' => 'choice', 'widget' => 'select', 'name' => 'voc_material',
  'value' => 3, 'label' => '...', 'options' => [ ['value'=>1,'label'=>'...'], ... ] ]
```

**Доступ к БД:** handler может выполнять свои процессы через общие безопасные
helper'ы; ядро не знает внутреннюю SQL-логику сущностей; SQL никогда не строится
из сырого request.

## 4. Имена полей и парсер

- Формат имени: `prefix_name`. Имя выбирает **базовую** сущность, и только её.
- Двоеточие (`mlt_x:prm_y`) упразднено полностью — не поддерживается ни на запись,
  ни на чтение. Legacy-код — археология смыслов, не compatibility contract.
- Уродливые составные имена (`mlt_x__prm_y`) запрещены: сложное поведение
  не живёт в имени колонки (см. §7).
- `field_parse()` возвращает `raw / kind / entity / name` без ветки modifier
  и без флага legacy.
- Структурные элементы ядра (не сущности): `id`, `rel_main`, `dep_*` —
  обрабатываются pipeline напрямую.
- **Первичный ключ — `id`.** `index` — археология старого кода: конфликтует
  со смыслом SQL index, требует вечных backticks. Новые таблицы — только `id`.
- **Неизвестный префикс — fail-fast** (ошибка схемы/конфигурации в dev/debug).
  Read-only fallback не проектируется: конфигуратор не позволяет создать поле
  несуществующей сущности, других путей появления полей нет. Старая БД
  под новым ядром не запускается.

## 5. Паспорт сущности

Сущность = функция `ent_xxx(): array` в `entities.php`, ноль аргументов,
возвращает паспорт. Registry собирается диффом `get_defined_functions()`
с проверкой через `ReflectionFunction`; ручного реестра нет.

Минимальный паспорт: `id`, `label`, `db` (правила DDL для конфигуратора),
`create` (параметры создания), `handlers`. Сущность описывает и runtime-,
и конфигураторное поведение.

**Создание структуры — декларативно.** Конфигуратор читает секцию `db` паспорта
и создаёт поле/таблицу через общий DDL-механизм. Отдельный create-handler
у базовых сущностей не нужен — он появится, только если декларативного
описания однажды станет недостаточно. Богатые admin-режимы не проектируются.

```
ordinary pipeline не вызывает create/admin modes;
configurator pipeline отдельно выполняет DDL по паспортам сущностей.
```

Добавление новой сущности = новый `ent_xxx()` + handlers. Правка ядра при этом
запрещена — если потребовалась, архитектура нарушена.

## 6. Пространство имён функций и файлы

Проектного префикса (`gpdp_`) нет: он не нёс информации — весь код здесь
и так GPDP. Имя функции первым словом отвечает «какому слою принадлежу».
Пространство формализовано: **закрытый словарь семейств + одно открытое
множество сущностей**.

| Семейство | Слой | Файл |
|---|---|---|
| `entity_*` | реестр сущностей | core.php |
| `field_*` | разбор имени, пакет, точка исполнения | core.php |
| `snapshot_*` | модель: сборка, хранение, lock | core.php |
| `action_*` | карта действие → режим | core.php |
| `record_*` | универсальная запись | core.php |
| `lookup_*` | инструменты сущностей | helpers.php |
| `render_*` | вывод | render.php |
| `ent_<id>` | паспорта | entities.php |
| `<id>_*` | функции сущности | entities.php |

Правило сущностей: **префикс функций сущности буквально равен префиксу
поля БД** (`voc_handler` ↔ колонка `voc_mr`) — Naming-Driven Behavior,
перенесённый в пространство функций. Это единственное открытое
(растущее) множество имён.

Словарь зарезервированных id (константа `ENTITY_RESERVED_IDS` в ядре):
имена семейств, классов и файлов не могут стать сущностями. Проверяется
при сборке реестра, не в горячем пути. Резерв классов на будущее:
`adm_` (конфигуратор), `chk_` (режим проверки), `fmt_` (конвертация).

**Файлы и направление вызовов** (стрелки не разворачиваются):

```
index.php  →  core.php  →  entities.php  →  helpers.php
index.php / результат конвейера  →  render.php
```

- config.php — окружение: БД, пути, параметры приложения; доступ через `config()`
- state/ — сгенерированное состояние (snapshot.php, schema.lock);
  код отдельно, порождённое отдельно; в git не коммитится (кроме .gitkeep)
- entities.php не командует index; helpers не знают про request;
  render не лезет в БД; core не знает предметную модель.

root_table — параметр модели; временно живёт в config.php до появления
конфигуратора, затем переедет в служебную таблицу модели.

## 7. Тонкая труба: проверки живут на границе, один раз

Система делится на зону подготовки и зону доверия. Каждая проверка
существует ровно один раз — на границе. Внутри конвейера данные
не перепроверяются, потому что пути, минующего границу, не существует
по построению. Дублирование проверки не добавляет безопасности при
единственном входе — оно добавляет второе место, которое можно забыть
обновить, и тем размывает источник истины.

```
грязный request
  → подготовка: action-карта, сверка со snapshot, сборка пакетов
  → подготовленное задание (prepared task)
  → тонкий конвейер: field_exec($data, $mode) без единой проверки
  → структурированный результат
  → render | запись | redirect
```

Не request → pipeline, а **request → prepared task → pipeline**.

Сложность из §-о-разрешённых-зонах разрешена в *подготовке* именно
затем, чтобы конвейер мог позволить себе быть тривиальным.
Толстая граница — тонкая труба.

Стандарт завершения write-цикла — **PRG**: POST → save → redirect → GET.
После успешной записи не отрисовка, а redirect на просмотр.

## 8. Schema snapshot

Runtime не интроспектирует живую БД (`SHOW TABLES` / `SHOW COLUMNS`) —
только bootstrap, явная пересборка, админ-режим.

Snapshot трёхслойный, пути обновления разные:

| Слой | Содержимое | Обновление |
|---|---|---|
| structural | таблицы, поля, типы, nullable/key, enum values, обратный граф dep_ | lock → DDL → rebuild → validate → unlock |
| presentation | подписи и параметры отображения (model_labels) | лёгкий refresh, без DDL-lock |
| model | адресное пространство и статусы (model_registry, §17), скомпилированные планы словарных проекций (§16) | лёгкий refresh, без DDL-lock; валидация по structural при каждой пересборке |

**Формат snapshot-файла — PHP:** `<?php return [ ... ];`. Загрузка через
`include`, opcache кэширует бесплатно, без json_decode. Тот же паттерн,
что entities.php. JSON — только для будущего экспорта/диагностики.

Атомарная запись: temp file + `rename()`. Режимы: locked (event-driven
regeneration) и normal (чтение кэш-файла).

**Служебные таблицы хранятся в snapshot как есть.** Строка подписей
(model_labels, §17) доступна целиком; нужную колонку выбирает потребитель
прямым обращением по ключу. Никакого маппинга `колонка → label`, никакой
обязательной `get_label()`-обёртки. То же правило для остальных служебных
таблиц: **система расширяется данными, а не новым API**.

**Снапшот всегда восстановим удалением.** Удаление snapshot-файла + любой
запрос = эквивалентная пересборка из трёх источников: DDL (физика),
реестр (адреса и статусы, §17), паспорта (поведение). Факт, живущий
только в снапшоте и нигде больше, — ошибка архитектуры. Следствия:
снапшот не входит в бэкапы, миграции и версионирование — переносится
модель, снапшот отрастает на месте. Формула: **реестр — исходный код
модели; снапшот — её проверенный и скомпилированный вид.**

Snapshot содержит параметры приложения, а не ядра:

```php
'application' => [ 'root_table' => 'main' ]
```

Корневая таблица — параметр модели. Ядро не знает слова «main».

## 9. Механизм расширения (вместо крючков)

Два уровня с разной ответственностью:

- **базовая сущность** отвечает за поле: правила хранения (декларативно, секция `db`),
  чтение, форма, валидация и нормализация;
- **сложный случай** (projected/filtered словари, отчёты, выборки, меню) отвечает
  за подготовленное представление и строится **поверх** базы.

Канонический путь расширения:

```
в код добавляется типовой обработчик;
в БД создаётся модельный элемент, который запускает этот обработчик
с уточняющей информацией.
```

`field_config` = модельные параметры конкретного элемента, физически живущие в БД
(служебные таблицы GPDP-конфигурации). Не внешний файл, не код.
Проектирование этого слоя отложено до работающей базы.

## 10. Запись и границы доверия

- `dep_*` / `rel_main` **не пишутся обычным save**. Универсальный save работает
  по whitelist полей, структурные исключены всегда. Изменение принадлежности —
  отдельное защищённое действие (`reparent` / `change_relation`).
- Составные input-поля (date = day+month+year; money, file, period) собирает
  и валидирует handler сущности / её типовой helper. Универсальная функция
  записи не знает устройства ни одной сущности.
- Идентификаторы (table/field) — только из snapshot + backticks;
  значения — только prepared statements. Не смешивать.
- Хранение multiple choice: стратегически m2m/junction table.
  CSV-в-колонке — временная storage strategy прототипа, не норма.

## 11. Обязательный минимум проверок (переходный режим hidden-полей)

До появления form contract (`_form_id → серверный контракт`) hidden-поля
допустимы при выполнении всех десяти проверок:

1. action только из whitelist;
2. action → внутренний mode через action_map;
3. table существует в известной модели;
4. field существует в известной модели;
5. field разрешён для текущего режима;
6. handler только из паспорта;
7. request не содержит callable function names;
8. ordinary route не имеет доступа к schema/admin actions;
9. dep/rel не пишутся обычным save;
10. значение проходит validate handler сущности.

## 12. Анти-паттерны (табу)

- модель данных или предметная область в коде ядра, включая пустые заготовки
  «на будущее» (`'report_modes' => []` и подобное);
- вызов функции по строке из request в любом виде;
- общий dispatcher для обычного и админ-контура;
- live introspection БД в рабочем режиме;
- двоеточие или составное поведение в имени колонки;
- HTML внутри handler'ов;
- сырой `$_POST`/`$GP` как аргумент handler'а;
- ООП-собор: ServiceLocator, Repository-per-entity, DI-контейнер, ORM-центр,
  Factory-слои. GPDP процедурен по духу;
- слой, не помогающий безопасно довести выполнение до handler call, — лишний.

## 13. Разрешённые зоны сложности

schema snapshot; lock при DDL; валидация паспортов; field parser; form contract;
проверки прав; валидация SQL-идентификаторов; правила конфигуратора.
Везде остальное достаточно: `$result = ($passport['handlers'][$mode])($data, $mode);`

Список не закрыт. Новый элемент допускается только если предложение прошло
все десять вопросов §15 **явно, по номеру пункта** — не общим ощущением
согласованности с духом системы. Легитимность расширения — не в том, кто
его предлагает и сколько понимания продемонстрировал в разговоре (это
не переживает смену сессии), а в том, что каждый пункт §15 назван и
закрыт письменно, проверяемо для любого читателя документа, включая
следующую сессию без контекста этого решения.

## 14. Финальный контракт базы (сводка)

```
Чтение/форма:  field name → entity passport → handler(data, mode) → structured result
Запись:        request → проверка action/table/field → validate handlers
               → normalized values → universal DB save → operation result
Словарь:       поле voc_x → закрытая лестница разрешения (§16):
               явная voc_x (склад | паспорт) ИЛИ адресный словарь
               x.id + x.data_name → скомпилированный источник
               → options / резолв id→label / валидация значения
Структура:     configurator → секция db паспорта → DDL engine → snapshot rebuild
```

Обычная форма не вызывает DDL и не может добраться до create/admin modes.

## 15. Контрольные вопросы к любому предложению

1. Не превращает ли GPDP в обычное приложение?
2. Не привязывает ли ядро к конкретной модели данных?
3. Не требует ли правки ядра при добавлении сущности?
4. Не смешивает ли request data и trusted dispatch data?
5. Не даёт ли обычной форме доступ к admin/schema?
6. Не заставляет ли runtime интроспектировать живую БД?
7. Не узаконивает ли плохой legacy-формат как стандарт?
8. Не плодит ли абстракцию там, где хватает passport handler call?
9. Сохраняет ли принцип «приложение = конфигурация GPDP»?
10. Может ли конфигуратор создать/изменить это без ручного программирования?

## 16. Словари: контракт потребления и паспорт проекции

**Словарь — не тип таблицы, а контракт потребления**: источник обязан
отдавать множество пар `id + label` (при необходимости — с контекстными
атрибутами) и уметь разрешать `id → label`. Всё, что исполняет контракт, —
словарь: таблица-склад значений, проекция обычной таблицы, срез
контекстной таблицы, системный реестр (снапшот, реестр паспортов).

Потребители контракта (все получают один механизм): форма (options),
чтение (id → имя), **validate** (словарь есть ограничение целостности:
значение сверяется с тем же источником и тем же фильтром, что и options),
выбор родителя для `rel_`/`dep_`, обратное разрешение имя → id при
миграции данных, фильтры и оси отчётов, селекты конфигуратора.

Инварианты:

1. **Разрешение словаря — закрытая лестница адресов** (пересмотр 2026-07-04:
   прежняя редакция требовала таблицу `voc_<имя>` всегда). Поле `voc_x`
   разрешается при сборке снапшота в строгом порядке:
   - таблица `voc_x` существует, форма склада → **простой словарь**;
   - таблица `voc_x` существует, маяк описателя → **паспортный словарь**
     (проекция по плану);
   - таблицы `voc_x` нет; таблица `x` существует и годится под простой
     словарь → **адресный словарь**: подпись наследуется от подписи
     таблицы `x` в реестре (§17);
   - `x` существует, но подпись собрать не из чего → fail-fast «таблице x
     нужен паспорт словаря»;
   - ни `voc_x`, ни `x` → fail-fast «неизвестный адрес».

   Явная таблица `voc_x` всегда побеждает адресацию. Адресация — не
   магия, а главный принцип (§1): имя поля есть явный адрес. Чего не
   существует ни на одной ступени — **эвристического выбора**
   (несколько кандидатов, система решает сама): кандидат либо один по
   построению, либо fail-fast. Словарные конструкции уровня паспорта
   создаёт конфигуратор по шаблону конструкции; неполный шаблон —
   ошибка сохранения, а не догадка системы.

   **Граница простое / сложное (уточнено 2026-07-11).** Простой словарь —
   это сходить в указанную таблицу и собрать подпись по шаблону из её
   СОБСТВЕННЫХ полей. Шаблон из одного поля (`data_name`) и из нескольких
   (`{voc_area} №{data_number}` → «Мамуринская №31») — оба простые:
   различие лишь в числе склеиваемых кусков. `data_name` перестаёт быть
   особым правилом «всегда бери эту колонку» и становится частным случаем
   шаблона из одного куска — одно правило вместо двух. Механизм склейки
   уже построен (составные подписи объекта, а2); простой словарь просто
   пользуется им, а не держит урезанный путь. Сложный словарь — когда
   добавляется что-то СВЕРХ этого: фильтры, срезы, контекстные условия,
   либо шаблон тянет куски из ДРУГИХ словарей (цепочка JOIN). Сложные
   подтипы описываются позже, отдельными решениями.

2. **Подтип словаря объявлен маяком.** Склад значений / паспорт проекции /
   будущие подтипы различаются явным признаком в самом словаре,
   не выводятся из структуры.

3. **Паспорт проекции — модель запроса данными, не SQL.** Строки паспорта
   называют источник и роли: таблица-источник, поле id, поля частей label,
   шаблон склейки, контекстные атрибуты, (позже) фильтр. Ссылки на чужие
   таблицы и поля хранятся значениями строк (`source_table`,
   `source_field`); точка и двоеточие в физических именах колонок
   запрещены — смысл не вшивается в имя спецсимволом. Текст SQL в данных
   не хранится и из данных не исполняется никогда: запрос собирает только
   код, по снапшоту и белым спискам.

4. **Компиляция — на сборке снапшота, ровно один раз.** Каждое имя
   паспорта сверяется со снапшотом при сборке; в снапшот кладётся готовый
   план запроса (модельный слой снапшота; обновляется без DDL-lock,
   валидируется по structural при каждой пересборке). Runtime исполняет
   план и не проверяет ничего — тонкая труба (§7). Осиротевший паспорт
   (исчезло поле/таблица источника) роняет пересборку fail-fast с именем
   паспорта.

5. **Проекция не добавляет новых элементов модели.** Части label —
   существующие поля, разрешаемые по имени; словарные части разрешаются
   компилятором в JOIN-цепочку по снапшоту, глубина цепочки ограничена
   только завершимостью: **цикл ссылок (паспорт достижим из самого себя) —
   fail-fast сборки с перечнем цепочки.**

6. Физическая запись паспорта — обычные строки обычной таблицы: паспорта
   переносятся дампом вместе с моделью, редактируются штатным CRUD
   (это не DDL: lock не нужен, нужен refresh модельного слоя снапшота).

Классификация перед выбором хранения (доктрина «О_словарях»): прежде чем
называть структуру «A + B + параметр» связью m2m, определить смысловую
роль — словарь / зависимая таблица / таблица состава или контекстного
применения / таблица, приводимая к словарю / проекция / отчётный срез /
настоящая m2m. Контекстная таблица хранит роль элемента и параметры его
применения и сама может быть источником производного словаря. Generic
junction — последний вариант, не первый.

## 17. Реестр модели: model_registry и model_labels

Модель живёт в служебных таблицах семейства `model_*`. Их формат ядро
знает как механизм (как формат снапшота и паспортов); содержимое — данные
модели. Служебные таблицы регистрируются в реестре наравне с предметными.

**model_registry — адресное пространство модели.** Каждый элемент —
строка: `data_kind` (`table` | `field`; новые kind вводятся решением, не
впрок), `data_owner` (таблица-владелец; NULL для kind=`table`),
`data_element` (имя элемента), `active`. Полный адрес уникален:
`(data_kind, data_owner, data_element)`. Запись в реестр — часть
структурной операции, внутри lock-окна:
`lock → DDL → реестр → rebuild → validate → unlock`.

**model_labels — подписи.** Зависимая таблица реестра
(`dep_model_registry`): подпись не живёт без элемента. Пара
`data_short` / `data_full`; контекст «таблица или поле» подписи не
хранят — он в `data_kind` реестра.

Границы — не размывать:

```
registry:  что это за элемент, где он находится, жив ли он
labels:    как его назвать человеку
passports: как он работает
```

Колонка реестра, описывающая «как обрабатывать элемент», — ошибка
проектирования: это строка паспорта.

**Реестр не хранит связи между элементами.** Граф отношений модели уже
записан в именах полей (`dep_`, `rel_`, адресация `voc_x → x`, §4 и §16):
имя *является* связью. Колонка реестра вида `data_relation` дублировала
бы этот граф вторым источником истины — запрещено по той же причине, что
и любое дублирование состояния (§12). Связи читаются разбором имени, как
и до появления реестра; реестр остаётся чистым адресным пространством
(что это, где, жив ли), не графовой СУБД.

**active** — структурное имя (ряд `id` / `dep_*` / `rel_main`),
допустимое в служебных таблицах модели; появление в предметной таблице —
ошибка модели. `active = 0` исключает элемент из model- и
presentation-компиляции (планы, формы, словари, отчёты); structural-слой
остаётся правдивым — физика не скрывается. Паспорт или словарь,
зависящий от неактивного элемента, роняет сборку fail-fast.

**Конвенция data_name.** Поле `data_name` — всегда просто «имя»;
субъектность (имя *чего*) несёт таблица-владелец. Таблица, адресуемая
словарём уровня 1 (§16), обязана иметь `data_name`; нет имени — нужен
паспорт словаря.

**Мусор реестра.** Строка, адресующая исчезнувший элемент, инертна:
компиляция её игнорирует и заносит в отчёт сборки (единственное место
аудита мусора). Зависимость плана, паспорта или словаря от мусора —
fail-fast с именем цепочки.

**Реестр сам исполняет словарный контракт** (§16): селекты конфигуратора
(«выбери таблицу», «выбери поле таблицы x») — словарные запросы к
реестру с фильтром по kind/owner, тем же механизмом, что предметные
словари.
